### Retro pixel-art rendering pipeline (Godot 4.7, gl_compatibility, GDScript, macOS, 1280x720, code-driven UI)
SUMMARY: This project already has three of the four pillars in place: rendering_method=gl_compatibility, default_texture_filter=0 (Nearest), stretch mode=canvas_items, and a 1280x720 base. What is MISSING and causes shimmer on movement is: snap-to-pixel (no 2d/snap keys in project.godot), an integer scale factor, and (for true retro density) a low-INTERNAL-resolution render pass via SubViewport + TextureRect that is then nearest-upscaled. The highest-ROI retrofit is additive and non-destructive: (1) add snap keys globally, (2) wrap the gameplay canvas in a SubViewport rendered at a low integer-divided resolution, (3) keep the existing code-drawn UI in a separate high-res layer so text/gauges stay crisp, (4) post-process the upscaled texture with a retro palette + slight CRT shader via canvas_item shaders (fully supported on gl_compatibility). On gl_compatibility specifically: SSR and 2D HDR do NOT exist at all in 2D regardless of renderer (they are 3D-only concepts), so do not chase them; canvas_item shaders, SCREEN_TEXTURE, SubViewport, TextureRect, StyleBoxTexture, CanvasItemMaterial, GPUParticles2D, and create_tween() are all fully available and are the correct tools here. The main gl_compatibility caveat is that some Forward+ canvas_item shader features (certain built-ins) differ and custom vertex data/extra UV channels are unsupported engine-wide.
TOP RECS:
  - Add the two snap-to-pixel keys to project.godot [rendering] (snap_2d_transforms_to_pixel=true, snap_2d_vertices_to_pixel=false) - 1 line, fixes most shimmer immediately.
  - Wrap the gameplay canvas in a 320x180 / 480x270 SubViewport and nearest-upscale via a TextureRect at integer 4x/2.7x scale; keep the existing code-drawn gauge/label UI in a separate high-res CanvasLayer so it stays crisp.
  - Adopt a single PPU convention (e.g. 16px = 1 tile) and reimport all sprites as Nearest/Lossless/no-mipmap; snap all draw_* coordinates with snappedf() in _draw().
  - Add one fullscreen canvas_item post-process shader (ColorRect above the TextureRect) using SCREEN_TEXTURE for retro palette quantization + light scanlines - the single biggest 'retro mood' lever for low effort.
  - For smooth camera motion without shimmer, render into the low-res SubViewport and tween the TextureRect position in high-res space rather than relying solely on global snap (works around known 4.3-4.7 snap-jitter bug).
PITFALLS:
  - SSR and 2D HDR do not exist in Godot 2D at all (3D-only concepts) - do not attempt to enable them; canvas_item SCREEN_TEXTURE + custom shaders are the correct 2D post-process path and are fully supported on gl_compatibility.
  - Custom vertex data and extra UV channels are unsupported in canvas_item shaders engine-wide; vertex colors are 8-bit. Pass extra data via instance uniforms (max 16) or texture atlases.
  - Setting snap_2d_transforms_to_pixel on a SubViewport can cause Tween/GPUParticles2D jitter when motion is non-integer (GitHub #98764, #110582) - animate integer positions or snap only the root viewport.
  - Using stretch/mode="viewport" (instead of "canvas_items") blurs nearest-filtered output under non-integer scales; keep "canvas_items" (already set) for crisp scaling.
  - Mixed-density art: leaving some sprites Linear while forcing Nearest globally will blur pixel art OR pixelate smooth vector art; set texture_filter per CanvasItem deliberately - the current code already draws smooth gradients/ellipses that you likely want to keep smooth.
  - Non-integer scale factors (e.g. 320x180 scaled to a non-720 height) cause directional shimmer; always pick low-res size = window_size / integer (320x180 for 1280x720 = clean 4x).
  - Some Forward+ canvas_item shader built-ins behave differently or are unavailable under gl_compatibility; test shaders on the target renderer, not just the editor default.
TECHNIQUES:
  - Enable global snap-to-pixel in project.godot [S]: Snaps all 2D transforms and polygon vertices to integer pixel boundaries, eliminating sub-pixel wobble/shimmer when sprites, cameras, or code-drawn primitives move.
  - Low-internal-resolution SubViewport + nearest-upscaling TextureRect [L]: Renders the gameplay at a low internal resolution (e.g. 320x180 or 480x270) then upscales crisply to 1280x720 with nearest-neighbor, giving chunky retro pixels and consistent pixel density while letting UI render at full 720p.
  - Per-node texture_filter override (mixed HD + pixel art) [S]: Lets pixel-art sprites stay Nearest while smooth-vector art (your existing gradients/ellipses) and UI fonts stay Linear-smoothed, so the retrofit does not force-blur currently-clean vector art.
  - Integer scale factor with canvas_items stretch + keep aspect [M]: Guarantees the internal-resolution-to-screen ratio is always a whole number so pixels never get unevenly stretched (the root cause of directional shimmer).
  - Retro palette swap via canvas_item shader (post-process) [M]: Recolor the upscaled low-res image into a constrained retro palette (e.g. 16-32 colors / GameBoy/Pico-8 style) and add optional scanline/CRT grain, all in one fullscreen pass.
  - Consistent pixel-density asset/import presets [M]: Ensures every imported sprite uses Nearest filtering, no mipmaps, no lossy compression, and a fixed PPU (pixels-per-unit) so art authored at one density never appears at the wrong scale.
  - Smooth camera over pixel grid (eliminate viewport jitter) [M]: Lets the camera move in smooth sub-pixel floats for organic motion while the visible pixels stay locked, fixing the classic Godot 4 'Snap 2D Transforms causes camera stutter' issue.

### Game feel & feedback (juice) for a 2D fishing JRPG — Godot 4.7, gl_compatibility renderer, 1280x720, code-driven GDScript UI (Control nodes, no AnimationPlayer scenes)
SUMMARY: All techniques here rely on three universally-available gl_compatibility features: (1) the SceneTreeTween API — create_tween(), Tween.set_trans(), Tween.set_ease(), and the four Tweener types (PropertyTweener, IntervalTweener, CallbackTweener, MethodTweener) — which are pure CPU/script and renderer-agnostic; (2) Control/Node2D transforms — position, scale, rotation, modulate, self_modulate — cheap on any renderer; (3) GPUParticles2D and CanvasItem shaders, both fully supported by gl_compatibility. What is UNAVAILABLE and must be avoided: 2D HDR is a no-op on this renderer (Godot docs confirm gl_compatibility always renders low-dynamic-range), so any \"brighter than 1.0\" bloom/tonemap trick in a canvas_item shader will clamp — design glow as additive modulate or screen-blend inside the shader instead; Screen-Space Reflections are Forward+ only (not needed for 2D anyway); compute shaders are unsupported (so no compute-based particles or GPU particle attraction fields — but standard emit/anim GPUParticles2D works fine); 2D glow/bloom environment post-processing has limited/none of the HDR-driven behavior. Core rule: drive every visible change through a tween with a non-linear transition, and layer multiple short tweens (squash + shake + popup + flash) so impacts stack. The existing gauge_bar.gd, fishing_screen.gd, and underwater_view.gd are the natural attachment points.
TOP RECS:
  - Squash & stretch on EVERY impact (fish bite, hit, button press). Highest ROI, ~15 lines, renderer-independent. Attach to gauge_bar.gd and a shared Juicer autoload. This alone transforms the feel.
  - Add a trauma-based screen shake on a dedicated CanvasLayer (ShakeLayer), not per-node. ~20 lines, reusable via add_trauma(amount). Use FastNoiseLite for smooth shake, cap at ~10px on 720p to avoid nausea.
  - Make gauges feel alive in gauge_bar.gd: exponential lerp toward target (framerate-independent via 1-exp(-k*delta)) + a slow ghost/trailing bar + a self_modulate pulse on damage. The trailing-bar technique is the single biggest gauge-quality upgrade.
  - Stack 3-4 cheap effects for the level-up moment: one-shot GPUParticles2D radial burst + ColorRect flash + TRANS_BACK scale punch on the label + mild shake + chord-synced audio. Stacking cheap effects reads as premium; one expensive effect reads as flat.
  - Wire every Button through a shared button-feedback script (hover 1.05, press 0.92 squash, release TRANS_BACK overshoot), always kill() the previous tween first. A menu-heavy JRPG lives or dies on button tactility.
PITFALLS:
  - 2D HDR is a NO-OP on gl_compatibility — it always renders low dynamic range. Do NOT try to get 'glow' by tweening modulate.a or energy above 1.0 and expecting bloom; it clamps to flat white. Fake glow with additive blend, a canvas_item energy shader, or GPUParticles2D sparkle.
  - Screen-Space Reflections, 2D environment glow/bloom (HDR-driven), and compute shaders are UNAVAILABLE on gl_compatibility. Standard GPUParticles2D and canvas_item shaders ARE available — use those instead.
  - set_trans() and set_ease() only affect Tweeners added AFTER them in the chain — they must be set before each tween_property()/tween_method(), not after. Calling them once at the start does NOT retroactively style earlier tweeners.
  - Control nodes scale from top-left unless pivot_offset is set to size*0.5 (in _ready, after size is known). This is the #1 reason squash/level-up punches look broken. Node2D scales from origin — offset position for centered scaling.
  - You CANNOT tween a shader uniform with tween_property('shader_parameter/energy', ...). Use tween_method with a Callable that calls material.set_shader_parameter(name, value).
  - Always store the tween reference and call kill() before starting a new tween on the same property (rapid button clicks, repeated hits) — otherwise tweens fight and the result jitters or stalls.
  - Don't use linear easing for anything organic. TRANS_LINEAR reads as robotic; use EASE_OUT with TRANS_BACK/TRANS_ELASTIC/TRANS_QUAD for the 'juicy' feel. Linear is acceptable only for steady gauge fills.
  - Hit-stop freeze will hang the game forever if the unfreeze tween/Timer is on a node that gets paused. Put the freeze-driver on a PROCESS_MODE_ALWAYS node, or use a one_shot Timer with process_mode=ALWAYS. Keep freezes <100ms.
  - Shaking each node individually is wasteful and drifts. Shake one CanvasLayer.offset wrapping the gameplay; optionally exclude the HUD layer so gauges stay readable.
  - On 1280x720, screen-shake amplitude and tween distances scale visually large — keep shake <= ~10px and popup rise ~40px, or the small viewport makes motion feel violent/nauseating.
TECHNIQUES:
  - Squash & stretch on impact (fight hits, fish bite, button press) [S]: Deforms a node's scale non-uniformly for ~80-150ms to give weight and elasticity to any impact — the single highest-ROI juice technique. Anticipation (pre-wind) + snap + settle makes fights feel alive.
  - Trauma-based screen shake (CanvasLayer.offset + FastNoiseLite) [S]: Adds camera/world shake on heavy hits, fish escapes, reel snaps. Trauma model (0..1, decays over time, amplitude = trauma^2) gives natural, non-nauseating, scaleable shake.
  - Hit-stop / freeze frame via process_mode + brief tween pause [M]: Freezes everything for 40-90ms on a big hit to sell weight (the classic fighting-game punch pause). Makes a fish catch or critical reel-in feel powerful.
  - Floating damage / number popups (Label spawn + rise + fade) [S]: Spawns a Label that rises, scales-in with overshoot, and fades — shows damage/stamina drain/coin gain and gives every number personality.
  - Gauge fill lerp + pulse (tension/stamina/energy bars — already in gauge_bar.gd) [M]: Makes gauges feel responsive: the displayed value lerps toward the target (smooth catch-up), the bar pulses/flashes on damage, and a ghost bar shows recent loss (the 'delayed trailing bar' look from fighting games).
  - Level-up burst (radial particle burst + scale punch + flash + sound sync) [M]: A celebratory multi-element payoff on level-up: radial GPUParticles2D burst, screen flash, a scale punch on the level label, and a chord synced to audio. Stacking 4 cheap effects reads as one premium moment.
  - Button press / anticipation feedback (press-down, hover breathe, release) [S]: Every Button responds: hover gently scales up, press squashes down (anticipation), release springs back. Turns dead UI into tactile UI — critical for a menu-heavy JRPG.
  - Additive glow / flash shader via CanvasItemMaterial (HDR-free) [M]: A canvas_item shader that fakes bloom/glow by adding energy and optionally desaturating — used for level-up bursts, critical fish flashes, and charging gauges. Works around the fact that 2D HDR is a no-op on gl_compatibility.
  - Audio-synced tween timing (hit on the beat / on the chord) [M]: Times a visual peak (scale punch, flash, particle emit) to land exactly when the audio waveform hits, so the moment feels scored rather than incidental.

  - Do not scale pixel art non-integer (e.g. 1.5x) — it destroys the pixel grid. If a fish must resize, re-author at the target cell size in Aseprite rather than scaling at runtime.
  - Original-art rule: 'inspired by' textual references are fine in the ledger, but never eyeball-replicate a specific existing sprite's pixels — even partial pixel-level copying violates docs/08. Record AI-assistance honestly if any generative tool touched a sprite.
  - macOS retina: allow_hidpi is fine, but confirm the window content_scale_size stays 1280x720 and integer scale is on, otherwise logical pixels become non-square and circles (fish eyes) render as ovals.
TECHNIQUES:
  - Configure global pixel-art project settings (nearest filtering + integer stretch) [S]: Locks every Texture2D/Sprite2D/TextureRect to crisp nearest-neighbor scaling and enforces integer viewport scaling so a 1280x720 internal resolution never produces shimmering sub-pixel fish/UI edges. This is the foundation that makes hand-authored pixel art read correctly on macOS retina displays.
  - Author fish/backgrounds/icons in Aseprite and import via Aseprite Wizard plugin [M]: Replaces the current code-drawn vector fish/backgrounds with charming, readable hand-pixelled art. Aseprite layers/tags become Godot SpriteFrames or AnimationPlayer tracks automatically, so swim/idle/hit/caught animations drop straight into AnimatedSprite2D without manual sheet slicing.
  - Build a texture atlas with AtlasTexture for UI frames and icons [M]: Packs all UI frame slices (panel corners, borders, button faces) and item/fish icons into one texture to cut draw calls and let a single CanvasItemMaterial/shader apply to the whole UI batch. Keeps the 1280x720 frame budget low and avoids z-fighting/overdraw from many loose Textures.
  - Per-texture import presets: disable mipmaps, set lossless VRAM, fix alpha [S]: Guarantees pixel-perfect small sprites (fish, icons, fonts) don't get blurred by automatic mipmaps or by lossy compression, and that semi-transparent water/foam edges stay clean. Pixel art at 720p never needs mipmaps and must stay lossless.
  - Fish readability system: silhouette-first authoring + hue/value contrast + sprite outline shader [M]: Makes fish instantly readable against busy underwater backgrounds — the single biggest charm/UX win. Uses deliberate palette discipline at author time plus an optional CanvasItemMaterial outline shader so fish 'pop' regardless of bg.
  - Scale rules & pixel-perfect placement at 1280x720 [S]: Establishes a deterministic scale vocabulary so fish, UI, and backgrounds stay crisp and proportional across the integer-scaled viewport. Defines the 'unit pixel' grid everything snaps to.
  - Canvas-item shaders for water/lighting without 2D HDR (canvas_item Light2D-style tint) [M]: Gives the underwater view depth gradient, caustics shimmer, and color tinting per-depth using cheap canvas_item shaders — achieving 'atmosphere' that the vector backgrounds currently lack, without relying on features gl_compatibility does not have.
  - Original-art licensing ledger per docs/08 (no copying existing works) [M]: Maintains a machine- and human-readable provenance record proving every Aseprite file is original art created for this project, satisfying docs/08's 'original art only, no pixel-for-pixel copying of existing works' rule. This is the compliance gate before shipping.
