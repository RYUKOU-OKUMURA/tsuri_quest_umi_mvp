# UI Screen Uplift（既存画面ブラッシュアップ）

## Purpose

既存画面を局所最適や微調整ループへ閉じ込めず、採用参照へ明確に近づけてfreezeする。正本は `docs/19_ui_production_playbook.md`（以下「プレイブック」）。

成果は「beforeより少し良い」ではなく、全画面の主役・読み順・状態別の収まり・操作性・質感が一体で改善したかで測る。作業順を **構成 → 主操作/主対象 → 情報階層 → 文字の収まり → 画像素材 → 細部演出** に固定する。

質感は共通キット/一点物の画像素材を正とする。絵画的・有機的な一点物はPNG、幾何フレーム・線画アイコンは共通キットのPNG/SVGを `Texture2D` として使う。runtime描画を質感の代替にしない。

## Load Before Acting

- `docs/19_ui_production_playbook.md` 全編。特に §0、§1/§1.1、§2、§3.4、§5、§6.4、§7。
- 対象画面の正式参照、またはspec/QAでユーザー採用済みと明記された最新モック。
- 対象画面のspec/UX契約と `docs/qa/<screen>_qa.md` のfreeze表・不採用リスト・微調整カウンタ・再検証TODO。
- 構成再設計、状態依存の余白、文字ずれ、最終仕上げを扱う場合は [references/convergence-pass.md](references/convergence-pass.md)。
- 新UIブロック追加なら `ui-screen-build` の分解・共通キット配線ゲートも併用する。

## Workflow

### 0. 画面タイプとupliftモードを決める

主役を画面タイプから定義する。メニュー/確認画面は主CTA、図鑑/一覧は選択対象、マップ/ゲームプレイはキャンバスやシーンが主役になり得る。港の「大型CTA」「1＋2」「2×3」を他画面へ型として移植しない。

プレイブック §1.1 で次を判定する。

- **局所uplift**: 主役・読み順は成立し、独立した1領域/1素材/1収束差分がTop1。
- **構成再設計**: 主役不在、同格要素、重複導線、過密と空白の同居、または差分Top3が同じ構成原因を持つ。

### 1. 現状と状態契約を固定する

`tools/<screen>_visual_qa.sh` で正確な1280x720実スクショを再生成する。無ければ最初に作る。古いキャプチャやOS UI入り画像で判断しない。

全状態の直積ではなく、**代表状態＋最も崩れやすい高リスク状態**を選ぶ。候補は任意領域の表示/非表示、0件/高密度、最長文字・最大値、ロック/解除、モーダル、選択状態。各状態について `状態 → 固定seed/データ → 出力名 → smoke契約` を対応させる。ユーザー提示画像の問題を直す場合は、その状態を可能な限り再現する。

### 2. 構成再設計ならモックを先に採用する

コードより先に、実データ密度を含む全画面モックを正確な画面寸法で作るか、候補から1案を選ぶ。方向性の選択が残る場合はユーザー承認を得る。モックをspec/QA/evidenceへ保存し、旧参照の何をsupersedeするか明記する。

採用モックを、画面タイプ、存在する/存在しない領域、主操作/主対象、補助操作、動的データ、状態表、固定アンカー、可変領域、主要矩形、素材/runtime分担へ分解する。関連する旧freezeだけを再オープンし、不関係なfreezeを維持する。

### 3. P1/P2/P3と差分Top3を決める

原寸と縮小比較から差分を分類する。

- **P1**: 見切れ・省略・重なり・素材未表示・読めない状態。
- **P2**: 全画面へ効く構成/素材差分、状態依存の不自然な空白、高視線領域の測定可能な光学不整列。
- **P3**: 根拠のない局所差、全画面で第三者が改善を判別できない好み。

P2/P3をpx数だけで分類しない。面積×視線優先度で「参照との差分Top3」をQAへ書く。共通キット未配線箇所も列挙する。

### 4. P1を処理する

維持する領域のP1を先に直す。採用モックで撤去する旧領域は旧構成を延命せず、置換後に「存在しないこと」とP1消滅をsmoke/スクショで確認する。

### 5. 1フェーズ1仮説で実装する

Top1に対し、変更仮説・動かす領域/値・不動値・採否条件を宣言する。

- 局所upliftでは独立した1差分だけを動かす。
- 構成再設計では、採用モックの情報階層仮説に必要な連動領域を同時に動かしてよい。
- 背景素材、配色、フォント、リファクタなど因果が独立する変更を同じ採否判断へ混ぜない。

### 6. 正しい画像素材経路を使う

- **キット部品**: `assets/showcase/common/` の既存PNG/SVG・9-sliceを先に配線する。無い幾何部品だけをPIL/SVG等で生成し、画面専用品にせずcommonへ追加/昇格してから使う。実使用の最小寸法で罫線・飾りが文字を横断しないことを確認する。
- **幾何小物**: セパレータや単色チップはプロシージャル生成可。
- **一点物**: 背景・キャラ・料理・魚・装飾モチーフ・大型演出はAI画像生成経路を使う。docs/12形式の仕様 → ソース保存 → 加工/統一処理 → 台帳記入まで行う。

生成した事実を採用理由にしない。現行へ全画面で明確に勝たなければ不採用にする。

### 7. 動的状態と操作状態を実装契約にする

任意領域が消えた空白は、由来と所有者を確認してから再配分する。消えた既存スロット由来なら可変領域へ渡せるが、empty state、将来のスクロール容量、heroの呼吸、シーン/マップの意味空間は自動で埋めない。主要操作と反復操作のアンカーを状態間で安定させる。

normal / hover / pressed / focus / selected / disabled のうち意味のある組合せを原寸で区別できるようにする。キーボード/ゲームパッドのフォーカス隣接も定義する。可変レイアウトは状態A→B→Aで元の矩形へ復帰することをsmokeで固定する。

### 8. 候補を同一状態で比較する

contact sheetが必要な素材は、ソース/処理後/現行/参照を確認してから既存スロットへ仮適用する。before/afterは乱数・動的データ・選択状態・viewportを一致させる。after/referenceのデータが一致しない場合は、読み順・密度・主役など比較する不変量を明記する。

採用条件は両方:

1. beforeへ全画面で明確に勝つ。
2. 対象Top1が縮小サムネイルで採用参照へ近づいたと第三者が判別できる。

### 9. 構成合格後に収束パスを1回行う

[references/convergence-pass.md](references/convergence-pass.md) の順で、余白配分 → 状態間アンカー → 文字の光学整列 → 最小寸法素材 → 操作状態を確認する。光学補正は描画画素/基準線を測定し、内部間隔を保ったクラスタ単位で行う。変更軸を限定し微調整カウンタへ記録する。明確に勝たなければrevertし、3回で別原因へ戻る。

最終判断では同一状態の原寸before/after、縮小、必要ならグレースケールを作る。結論を伝えず、原寸証拠を含めて独立レビューを受ける。greenのsmokeだけで見た目を合格させない。

### 10. 検証・記録してfreezeする

`./tools/validate_project.sh` と該当smoke/フロー検証を実行する。QAドキュメントのfreeze表を上書きし、不採用案・微調整カウンタ・直近判断ログを更新する。状態対応表と判断根拠を残し、比較画像を `docs/qa/evidence/<screen>/` へ保存する。P3だけが残ったら終了する。

## Acceptance Gate

- 画面タイプに応じた主操作/主対象が最初に読め、同格化・重複導線がない。
- 構成再設計では、採用モックの分解・関連freezeの再オープン・新UX smoke契約が記録済み。
- 代表状態＋高リスク状態でP1がゼロ。不自然な偶発空白がなく、意図した空白は守られ、主要アンカーが安定している。
- before/afterのseed・動的データ・選択状態・viewportが一致している。
- normal / hover / pressed / focus等の必要状態が原寸で識別でき、A→B→A復帰を検証済み。
- 共通キット/一点物の画像素材が配線され、最小使用寸法で文字・アイコン干渉がない。
- 変更がbeforeに明確に勝ち、縮小比較でも対象Top1が採用参照へ近づいている。
- 独立レビューが原寸証拠を確認し、P1/P2の未解決を指摘していない。
- validate、該当smoke/フロー、素材参照監査がgreen。
- QAとevidenceが最新で、supersededログや `/tmp` のみの証拠が残っていない。

## Hard Rules

- 構成再設計ゲートが発火した画面を、旧配置の局所調整だけで収束させない。
- 方向性が未決の全画面再設計を、ユーザー採用モックなしで実装しない。
- 「空いている」だけで領域を埋めない。空白の役割と所有者を分類する。
- 状態の違うbefore/afterを採用根拠にしない。
- 1フェーズ1仮説を、無関係な変更を束ねる口実にしない。
- 測定なしの数px調整を繰り返さない。光学補正は構成合格後の収束パス1回に限定する。
- freezeはP1、ユーザー承認済み方向更新、測定可能な高視線収束差分以外で再オープンしない。
- 既存キット部品を `StyleBoxFlat` / `draw_rect` / PIL / SVGで画面ごとに再発明しない。不足する幾何部品はPIL/SVG等で生成してcommonへ昇格してよい。有機的一点物をSVGやプロシージャル描画で代替しない。
- beforeとの微小勝ち、パーツ単体の良さ、生成コストを採用理由にしない。
- 日本語テキストを画像素材へ焼き込まない。
- 新規ハードコードhex、他画面素材フォルダの直接参照、全画面フォント一括置換を行わない。
- 証拠を `/tmp` にしか残さずfreezeしない。
