# 画面別QAドキュメント運用規約

Date: 2026-07-03

docs/19 §7（記録規約）の下位規約。矛盾したら docs/19 を優先し、docs/19 側を改訂してから作業する。

旧 `design-qa.md`（水中ファイト）が英文・追記専用の約130箇条／5万トークン級ログになり、次の作業者が freeze値・不採用理由を抽出するコストが激増した反省から、書式を規定する。

## ファイル構成

- **1画面1ファイル**: `docs/qa/<screen>_qa.md`（例: `underwater_fight_qa.md`）。他画面の記録を混ぜない。
- **記述は日本語**。
- 過去の詳細経過は `docs/qa/archive/` へ移す（正本のQAファイルには残さない）。
- 採用判断の根拠にした比較画像は `docs/qa/evidence/<screen>/` へ日付付きファイル名でコピーする（例: `2026-07-03_full_compare.png`）。`/tmp` はセッションで消えるため、`/tmp` のみに残った証拠は検証不能として扱う。

## テンプレート

新しい画面のQAファイルは以下の構成で作る。各セクションの役割を変えない。

```markdown
# <画面名> QA判断ログ

最終更新: YYYY-MM-DD / 状態: v1 freeze中 など
参照画像: reference/<file>.png
QA更新コマンド: ./tools/<screen>_visual_qa.sh

## 1. freeze値（正本）

現在有効な値だけを書く。値を更新したら該当行を**上書き**する（追記して古い行を残さない）。

| 項目 | 値 | 場所 | 理由・備考 |
|---|---|---|---|

## 2. 不採用・再試行禁止リスト

同じ案の再試行を防ぐ。履歴を全読みしなくても済む粒度で書く。

| 案 | 却下理由 | 判断日 |
|---|---|---|

## 3. 微調整カウンタ

同一パラメータの微調整回数を記録する。3回で改善しなければ素材フェーズへ送り、行をクローズする（docs/19 §0 原則3の運用化）。
runtime描画の装飾追加（wash・罫線・影・StyleBoxFlat等）は、新しい装飾名でも「装飾パス累計」行で画面単位に合算して数える。累計3パスを超えたら素材フェーズへ送る（docs/19 §5-13）。

| パラメータ | 回数 | 直近の変更内容 | 状態 |
|---|---|---|---|
| 装飾パス累計 | 0 | — | — |

## 4. 暫定判定・再検証TODO

ランタイムキャプチャ不能環境での静的合成による暫定合格などを集約する。復旧後に再判定して消す。

## 5. 現在の残ギャップ

P2/P3の**現在の状態のみ**を書く。解決したら削除する。パッチ履歴の再説明をしない。

## 6. フェーズスコープ宣言（作業中のみ）

P2フェーズ着手時に「今回動かすパラメータ」「触らないfreeze値」を書く。フェーズ収束時に判断ログへ転記して消す。

## 7. 判断ログ（直近パスのみ）

最新パスについて「何を変えたか / 何を変えていないか / どの比較画像で判断したか / 採用・却下理由 / 同じループへ戻らないための固定条件」。
上書きされた（superseded）過去ログは削除し、残したければ archive へ移す。
```

## 禁止事項（docs/19 §9 と対応）

- 追記専用で肥大化させる（superseded の記録を消さない）
- freeze値を長文の中に埋める（必ず表で書く）
- 1ファイルに複数画面の記録を混ぜる
- 証拠画像を `/tmp` にしか残さない
- 微調整カウンタを付けずに同一パラメータを触り続ける
