# 36. サメ餌魚の釣行中UX改善（演出・表示レイヤーのみ。E10-4消費仕様は不変）

作成日: 2026-07-07
状態: 実装済み（2026-07-07）。ただし本docの『E10-4消費仕様は不変』という前提は docs/37・docs/38 で上書きされた
位置づけ: E10（`docs/v2/E10_shark_raising.md`）完了後のフォローアップ。危険海域の釣行体験を「餌魚の物語が見える」ものにする。**抽選重み・消費ルール・メガロドン条件には一切触らない**。

## 0. 設計原則（本docの判断基準）

このゲームの主ユーザーは子供。泳がせ釣りの核心体験は「**自分が釣った魚で、もっとデカい魚を釣る**」という因果の実感である。E10-4のメカニクス（港でセット→釣行開始時に消費→重み補正→サメ出現）は完成しているが、**中間過程が全部不可視**のため、子供の目には「アジがどこかに消えて、なぜかサメが出た」に見える。本docの改修はすべて「餌魚が仕事をしている瞬間を見せる」ことに奉仕する。在庫管理情報（残数など）の充実は子供の体験に寄与しないため扱わない（§1の不採用表）。

## 1. 背景・課題

危険海域（`danger_reef`）ではE10-4の泳がせ仕掛けにより、出港前にセットした餌魚（釣った魚1匹）を釣行開始時に消費してサメを誘う。しかし釣行中の画面では:

1. **餌魚の実名がほぼ見えない**。左下「使用中のエサ」パネル（`src/ui/components/fight_hud.gd` の `_rig_name_text()` / `_rig_bait_text()`）は仕掛け名と対応餌カテゴリ（「泳がせ／対応餌：小魚」）のみ。実際にセットした魚名は右上の釣り場詳細1行（`src/ui/fishing_screen.gd` `_spot_detail_text()` → `_shark_lure_summary_text()`）に埋もれるだけ
2. **魚影カード下部のテキストが枠で切れる**。`src/ui/components/fight_sidebar.gd` の未確認カード描画（`_draw_detail_line` 2行、L205-206「釣り場：〜」「タナ：〜 / エサ：〜」）がカード下端で縦に見切れる（2026-07-07 スクショで確認）
3. **餌魚とサメ出現の因果が語られない**。アタリ・ヒット・キャッチのどの瞬間にも餌魚が登場しないため、「港でアジをセットした」ことと「サメが釣れた」ことが子供の中で繋がらない
4. **好物システムが完全に不可視**。`SHARK_DIETS`（好物一致で重み×3.0、`GameData.is_favorite_food()` / `shark_lure_weights()`）は体験上存在しないのと同じ。子供が「次はイワシを試そう」と実験を始めるための手がかりがゼロ

### 検討済みで採用しなかった案（再検討時の前提）

| 案 | 不採用理由 | 判断日 |
|---|---|---|
| 餌魚のストック制（1投ごとに1匹消費・釣行中に残数表示） | 釣行中の在庫管理は「ワクワクの最中に算数をさせる」ことになり、子供の没入対象（ウキとサメ）から注意を奪う。ボウズで複数匹失う損失体験は子供に強烈に刺さり「もう危険海域に行かない」を招くリスク。加えてE10-4のリスク設計・メガロドンのヒットごと8%抽選が釣行単位前提で調整済みであり、決定#6「消耗品システム非導入」の例外を最小に保つ方針にも逆行 | 2026-07-07 |
| 釣行中HUDへの「餌魚残数」表示 | 現行仕様では釣行中に残数が変化しないため無意味な静的表示。残数が意思決定に効くのは出港前で、港画面の餌魚セットボタンが既に「魚名 x所持数」を表示済み（`src/ui/harbor_screen.gd` `_refresh_preparation_card()` L233） | 2026-07-07 |
| 仕掛け状態インジケータの新設 | 状態は既に「今の状況」パネルと魚影カードの「反応」タグの2箇所で表現済み。3箇所目はパネル増殖で情報階層を壊す | 2026-07-07 |

2026-07-07: この2件は docs/38 のレア度耐久チャージとして前提を変えた上で再採用された。再検討の経緯は docs/37

## 2. スコープ

### フェーズ1: 土台（見える化）

#### 改修A: 「使用中のエサ」パネルに餌魚の実名を表示

- 対象: `src/ui/components/fight_hud.gd`（描画）、`src/ui/fishing_screen.gd`（trip_stats受け渡しの確認のみ。`shark_lure_fish_id` / `shark_lure_fish_name` は `_trip_stats` に格納済み）
- 仕様:
  - 危険海域かつ餌魚セット時のみ、パネル下段の対応餌カテゴリ行（`_rig_bait_text()`）を「餌魚：<魚名>」表示に差し替える（例: 「餌魚：マアジ」）。上段の仕掛け名（「泳がせ」）は現状維持
  - 餌魚未セット時・通常釣り場は現行表示のまま。表示分岐は trip_stats の `spot_id == "danger_reef"` かつ `shark_lure_fish_id` が空でないことを条件にする（釣行継続・釣り場変更で payload が残っても通常釣り場に誤表示しない）
  - 魚ポートレートを添える場合は `FightFishAssets` 経由で取得（直接パス参照禁止）。まず文字だけで v1 とし、ポートレートは収まり確認後の任意加点
  - 日本語テキストのPNG焼き込み禁止（runtime描画のみ）
- 意味論: 表示は「この釣行で使った餌魚」であり在庫を意味しない（E10-4で釣行開始時に消費済み）

#### 改修B: 魚影カード下部のテキスト見切れ修正

- 対象: `src/ui/components/fight_sidebar.gd` の未確認魚影カード（`_draw_detail_line` 2行、`desc_y + 22.0` / `desc_y + 38.0` 付近）
- 仕様: 「釣り場：〜」「タナ：〜 / エサ：〜」の2行がカード内枠（`inner`）の下端からはみ出さないこと。`desc_y` はシグナルアート矩形（`signal_rect`、`rect.size.y` 比率で伸縮）からの積み上げで決まるため、(a) シグナルアート比率の圧縮、(b) 説明文 `_draw_wrapped` の行数制限、(c) 詳細行のフォント/行間圧縮のいずれかで吸収。**方針選択は実スクショで全状態（準備中/探り/ヒット、フレーム素材あり/なし）を確認して決める**

### フェーズ2: 餌魚の物語（子供体験の本丸）

フェーズ1完了後に着手。どちらも**演出レイヤーのみ**で、`fishing_simulator.gd` の抽選・状態遷移ロジック、`shark_lure_weights()`、消費処理には触らない。

#### 改修C: アタリ・ヒットの瞬間に餌魚を主語にする

- 目的: 「僕のアジがサメを呼んだ」という因果を、緊張が最大の瞬間（アタリ〜ヒット）に体感させる
- 対象: `src/core/fishing_simulator.gd` のメッセージ文言（`_set_message()`、APPROACH: 「魚影がエサへ近づいている……」/ BITE: 「食いついた！ E / Enterでアワセ！」）、`src/ui/components/fight_sidebar.gd` の未公開時「今の状況」カード（APPROACH/BITEで実際に見える文言）、`src/ui/fishing_screen.gd`（餌魚名の受け渡し確認）
- 仕様:
  - simulator に釣行コンテキストとして餌魚名（表示専用の String。空なら無効）を渡す。渡し方は既存の `player_stats` Dictionary への相乗りか専用プロパティのどちらでもよいが、**simulator が PlayerProgress / trip_stats を直接参照する形は禁止**（docs/26 R3 の pure 境界維持）
  - APPROACH/BITE中は `fishing_screen.gd` の上部メッセージパネルが非表示になるため、プレイヤーに見える主表示は `fight_sidebar.gd` の「今の状況」カード側にも反映する。simulator文言だけを変えて完了扱いにしない
  - 餌魚セット時のみ、メッセージを餌魚主語に差し替える。例:
    - APPROACH: 「魚影が**餌のマアジ**へ近づいている……」
    - BITE: 「**マアジに何かが食いついた！** E / Enterでアワセ！」
  - **正体秘匿の原則を守る**: アワセるまで魚種は明かさない既存設計（fight_sidebar「アワセるまで魚の正体は分からない」）に合わせ、餌魚名は出すがヒットした魚の名前・サメであることは出さない
  - 餌魚未セット時・通常釣り場は現行文言のまま
  - 文言は上記を基準に実装時に音読して調整してよいが、「餌魚の名前が主語になる」ことだけは外さない

#### 改修D: 好物発見のフィードバック

- 目的: `SHARK_DIETS` を「攻略情報」ではなく「子供が自分で見つける法則」にする。釣果の瞬間に1行のヒントを返すことで、「次は違う魚を試そう」という実験のループを起動する
- 対象: `src/ui/components/catch_fanfare.gd`（釣果演出。`_catch_result` に表示を追加）、`src/ui/fishing_screen.gd`（catch_result への好物一致フラグの詰め込み）
- 仕様:
  - 釣れた魚がサメ（`fish.get("shark", false)`）かつ餌魚セット釣行のとき、`GameData.is_favorite_food(shark_id, bait_fish_data)` を判定し、一致していればファンファーレに発見行を1行追加する。例: 「**ホシザメはマアジが大好物みたいだ！**」
  - 好物**不一致**のサメ釣果には何も出さない（「好物ではなかった」等のネガティブ表示は、餌魚選択を間違い扱いにするため禁止。無言＝手がかりなし、が正しい実験フィードバック）
  - 大型2種（shumokuzame / hohojirozame）は好物セット時のみ出現する仕様（E10-4）のため、釣れた時点で必ず好物一致。同じ文言系で構わないが、初回釣果はE10側の演出が主役なので**発見行が既存演出を邪魔しない位置・タイミング**にする
  - メガロドンは対象外（好物述語ではなくヌシ級述語＋Lv50＋全なつき度100の特殊条件。発見行を出すと誤学習させる）。実装では `fish_id != "megalodon"` を明示条件にする
  - 判定は pure 関数（`is_favorite_food`）の呼び出しのみで行い、新しい判定ロジックを UI 側に書かない
  - `catch_result` には表示済み文言（例: `favorite_bait_discovery_text`）または表示に必要な最小キーを詰める。`catch_fanfare.gd` 側で `GameData.is_favorite_food()` 相当の判定を再実装しない
  - 既存ファンファーレのボーナス欄は最大4行（`_bonus_label.max_lines_visible = 4`）。発見行は「称号獲得」より上、「自己記録更新」「撃破報酬」より下に差し込む。4行を超える場合は称号表示を最大1件に圧縮し、記録更新・撃破報酬・好物発見を優先する
  - 発見の記録永続化（図鑑への好物欄追加など）は本docのスコープ外。反応が良ければ別フェーズで検討

### スコープ外（やらない）

- E10-4の消費仕様・抽選重み・メガロドン条件の変更（§1の不採用表を参照）
- 港画面の餌魚セットUI変更（残数表示は実装済み）
- 図鑑への好物情報の永続表示（改修Dの反応を見てから別フェーズ）
- 右上の釣り場詳細「餌魚:○○」行の削除（冗長だが無害。改修A定着後に情報整理する場合は別パス）

1. **brief A（改修A）**: `fight_hud.gd` の餌魚名表示。触ってよいファイル: `fight_hud.gd` のみ。DoD: 危険海域・餌魚ありのスクショで「餌魚：<魚名>」が視認でき、通常釣り場・餌魚なしで現行表示が変わらないこと
