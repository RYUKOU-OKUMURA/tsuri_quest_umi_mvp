# 21. ステータス画面 v1 実装プロセス振り返り

Date: 2026-07-02

対象作業: `reference/08_status_screen_mockup.png` を基準に、現行 `src/ui/status_screen.gd` を本番寄りのステータス画面 v1 へ置き換える作業。

使用手順: `skills/ui-screen-build/SKILL.md` と `docs/19_ui_production_playbook.md`

## 結論

`ui-screen-build` の流れは今回かなり有効だった。特に、実装前に `docs/20_status_screen_spec.md` で参照画像を分解し、存在する領域/存在しない領域/データ責務を明文化したことで、旧 `status_screen.gd` の「プレイヤー・魚図鑑・所持品を横並びに詰める」構成へ戻らずに済んだ。

一方で、今回の画面は「完全新規画面」ではなく既存 `status_screen.gd` の置き換えだったため、`ui-screen-build` の言葉どおりに進めると少しズレる箇所があった。次回以降は「新規画面」と「既存簡易画面を正式UIへ昇格する作業」を同じスキル内で明示的に分けるとよい。

## うまくいったところ

### 1. 分解ドキュメントが巻き戻りを防いだ

実装前に `docs/20_status_screen_spec.md` を作り、参照画像の領域を次のように分けたことが効いた。

- 上部: タイトルと `PlayerStatusBar`
- 左: プレイヤー能力、EXP、装備、船、次の食事効果
- 中央: 釣果サマリー、最近釣れた魚、図鑑コンプリート率
- 右: クーラーボックス、所持竿、料理/食事効果ログ
- 下部: `魚図鑑`、`料理・食事`、`港へ戻る`

これにより、現行画面にあった大きな魚図鑑グリッドを磨き続けるのではなく、`fish_book` 画面へ責務分離する判断が早くできた。

### 2. 「存在しない領域」の明記が効いた

今回の重要な判断は、ステータス画面に全魚種一覧を残さないことだった。`docs/20` に「全魚種を並べる大きな魚図鑑グリッドは存在しない」と書いたため、実装時に旧 `FishBookGrid` 相当を撤去できた。

これは `ui-screen-build` の「参照に存在しない要素を実装しない」というルールがうまく働いた部分。

### 3. visual QAを先に作ったことで判断が安定した

`tools/status_visual_qa.sh` と `build_screen_visual_comparison.py` の `status` preset を先に追加したことで、実装後すぐに `/tmp/tsuri_status_compare.png` で参照と横並び確認できた。

コード上では成立していても、実スクショでは船名や食事効果に省略が出た。これはvisual QAがなければ見落としやすいP1だった。

### 4. smoke testで新UX契約を固定できた

`tools/status_smoke.gd` では、見た目ではなく構成契約を検証した。

- `PlayerStatusBar` がある
- 3ペインがある
- `魚図鑑` / `料理・食事` / `港へ戻る` が遷移する
- 旧魚図鑑グリッドが残っていない
- 釣果集計値がseedデータと一致する

これにより、次回の修正で旧構成が復活するリスクを下げられた。

### 5. 共通キット優先の判断ができた

ステータス画面専用PNGを先に作らず、`common/`、`PlayerStatusBar`、`make_return_button()`、`HarborBackdrop`、魚ポートレート共有素材でv1構成を作った。

結果として、素材生成に入る前に構成・導線・データ責務を固定できた。これは `docs/19` の「構成 → 主導線 → 情報階層 → 文字の収まり → 素材の質感」の順番に合っていた。

## うまくいかなかったところ

### 1. スキル名の「新画面ゼロイチ」と実態が少しズレた

今回の対象は新規画面ではなく、既存 `status_screen.gd` の正式化だった。そのため、`ui-screen-build` をそのまま読むと「新しい `src/ui/*_screen.gd` を追加する」方向に寄りやすい。

実際には、`main.gd` の `status` ルートも `tools/status_preview.tscn` も存在していたため、作業は「既存簡易画面の置き換え」だった。ここはスキル側に明示があると迷いが減る。

### 2. 参照画像に含まれる重複情報の扱いに判断が必要だった

参照画像には Lv/所持金が上部と左パネルに重複して見える。しかし `docs/19` では同一情報の重複表示が禁止されている。

今回は `docs/19` を優先し、上部 `PlayerStatusBar` に Lv/装備/所持金を集約し、左パネルでは所持金を出さない判断をした。この判断を `docs/20` に明記したのはよかったが、AI生成の完成イメージを扱うときは「参照画像のどこをそのまま採用しないか」を毎回明示する必要がある。

### 3. preview seed が最初は弱かった

初期の `tools/status_preview.gd` は数匹だけを発見済みにする簡易seedだったため、右ペインが空欄中心になり、完成イメージに近い密度を確認できなかった。

後から Lv、所持金、竿、船、クーラー在庫、料理ログ、釣り場別記録をseedする形へ更新した。今後はvisual QAを作る時点で、参照に近い「密度テスト用seed」を先に設計するべき。

### 4. 文字省略P1が実装後に出た

船名と次の食事効果で省略が発生した。原因は、左パネルへ長い説明を入れすぎたこと。

修正では、左パネルは短い状態名だけを出し、詳細説明は右ログや料理画面側へ寄せた。これは、参照画像へ寄せるよりも情報設計を優先した判断だった。

### 5. v1とv2の境界を途中で意識する必要があった

参照画像の外周木枠や真鍮感は、現状の共通キットだけでは完全には届かない。ここで専用PNG生成へ進むと、構成フェーズと素材フェーズが混ざる。

今回は専用PNG生成をv2候補として残したが、最初から「v1では共通キットで構成固定、専用素材はv2」と明文化しておくと、より迷いが少なかった。

## スキルへの改善提案

`skills/ui-screen-build/SKILL.md` には、次の補足を入れるとよい。

1. **既存簡易画面を正式UIへ昇格する場合の分岐**
   - 新規ファイル追加ではなく、既存 `*_screen.gd` の責務整理として扱う。
   - 既存previewや既存routeがある場合は再利用する。

2. **AI生成参照画像の非採用点を書くルール**
   - AI生成画像は情報重複や実装上の矛盾を含むことがある。
   - 参照画像から「採用しない要素」も分解ドキュメントへ書く。

3. **preview seed設計をvisual QA作成時に含める**
   - 空データだけでなく、参照密度に近いseedを用意する。
   - 所持品、ログ、発見済み/未発見、装備中などを含める。

4. **v1/v2境界の明文化**
   - v1は構成、主導線、情報階層、文字収まり、最低限の共通素材。
   - 専用PNG生成や看板品質の質感追求はv2候補として分ける。

## 次回同種作業の推奨手順

1. 参照画像を保存する。
2. 既存画面か新規画面かを最初に判定する。
3. 分解ドキュメントで、存在する領域/存在しない領域/採用しない参照要素を書く。
4. preview seedを参照密度に合わせて設計する。
5. visual QAを先に通す。
6. smokeで新UX契約を固定する。
7. 構成フェーズを共通キットだけで完了させる。
8. スクショでP1を潰す。
9. 専用素材が必要ならv2として別フェーズにする。

## 今回の最終判定

`ui-screen-build` は今回の作業に対して有効だった。特に「分解してから作る」「visual QAを先に作る」「存在しない要素を消す」という部分が強く効いた。

改善点は、スキルを「完全新規画面」だけでなく「既存簡易画面の正式化」にも自然に使えるようにすること。今回のステータス画面は後者だったため、その分岐が明文化されていればさらに進行が滑らかだった。
