## 0. 結論：最大ギャップは1本に集約される

現状は**「滑らかなベクター/ペイント調 ＋ 平面的な StyleBoxFlat ＋ システムフォント」**。
ターゲット（`docs/00` の「明るい海・レトロピクセルアート・和製RPG風UI」と参照モックアップ）は
**「レトロピクセルアート ＋ JRPGの9スライス装飾枠（ウィンドウスキン） ＋ ビットマップ/ピクセルフォント ＋ 演出（juice）」**。

つまり**美術方向の転換**が最大の差であり、同時に**5つの“共通レバー”**で全画面を一括引き上げできる。

## 1. ターゲット品質を構成する5要素

1. **ピクセルアート化** — 低解像度で描画し NEAREST で整数拡大（チャンキーで意図的なピクセル）
2. **JRPG 装飾枠（9スライス）** — 金縁・ベベル・角の宝石/鋲・二重線の「ウィンドウスキン」
3. **ピクセルフォント** — アウトライン＋落とし影付き、HUDの数値/魚名/深度まで全部レトロに
4. **統一パレット（治理）** — 16〜32色の固定インデックスで散在する hex を一元化
5. **演出（juice）** — スクイーズ＆ストレッチ、画面揺れ、ヒットストップ、フローティング数字、ゲージの追従/点滅

## 2. 共通レバー（全画面を一度に引き上げる5つの共有コンポーネント）

これらは**1回の変更で全7画面に効く**ため最優先。

| # | レバー | 主なファイル | 効果 |
|---|---|---|---|
| A | **ピクセルパイプライン** | `project.godot` + 新規 `SubViewport` | shimmer 解消・ピクセル密度統一 |
| B | **テーマ 9スライス化** | `src/ui/ui_theme.gd`・`screen_base.gd` | 「プロのJRPG」の最大シグナル |
| C | **ピクセルフォント** | `src/ui/ui_theme.gd` | 最も目立つ“レトロでない”箇所を解消 |
| D | **統一パレット** | 新規 `assets/themes/palette.gd` | 色のリズム・将来のパレット差し替え容易に |
| E | **Juice 基盤** | 新規 `autoload/juicer.gd`・`gauge_bar.gd`・`screen_base.gd` | 操作感と演出の格上げ |

### A. ピクセルパイプライン（`project.godot`）
```ini
[rendering]
2d/snap/snap_2d_transforms_to_pixel=true      # shimmer の主因を解消
2d/snap/snap_2d_vertices_to_pixel=false       # 回転/テアが問題の時だけtrue

[display]
window/stretch/mode="canvas_items"            # 既存（維持）
window/stretch/aspect="expand"                # 既存。真の整数拡大は下記SubViewportで
```
> 本物のチャンキーピクセルは**低解像度 SubViewport（320×180 または 480×270）を NEAREST で整数拡大**して実現。
> その際、HUD（ゲージ/文字/ボタン）は**別の高解像度 CanvasLayer に分離**して文字はクリアなまま残す。
> ※ `scale_mode="integer"` は Godot 4 の stretch 設定には無い。整数拡大は SubViewport 方式で行う。

### B. テーマ 9スライス化（最重要）
`StyleBoxFlat`（塗り＋1色枠＋角丸＋影）→ **`StyleBoxTexture` 9スライス**。`ui_theme.gd` に `_tex_stylebox()` を新設し、既存の `set_stylebox` スロットをそのまま置換。`DarkPanel/BluePanel` パターンを拡張して `JRPGPanel/JRPGHeader/JRPGDialog/JRPGButtonGold` を追加。
```gdscript
static func _tex_stylebox(tex: Texture2D, margin: int, tile_center := false) -> StyleBoxTexture:
    var sb := StyleBoxTexture.new()
    sb.texture = tex
    sb.texture_margin_left = margin; sb.texture_margin_top = margin
    sb.texture_margin_right = margin; sb.texture_margin_bottom = margin
    sb.axis_stretch_horizontal = StyleBoxTexture.AXIS_STRETCH_MODE_TILE_FIT
    sb.axis_stretch_vertical = StyleBoxTexture.AXIS_STRETCH_MODE_TILE_FIT
    sb.texture_filtering = CanvasItem.TEXTURE_FILTER_LINEAR_WITH_MIPMAPS  # ← 重要：既定NEARESTだと装飾が崩れる
    return sb
```

### C. ピクセルフォント
`build_font()` の `SystemFont`（Hiragino Maru Gothic）→ ライセンス確保済みの **JP対応 FontFile**。
```gdscript
var f := FontFile.new()
f.font_data = load("res://assets/fonts/jrpg_pixel_jp.ttf")
f.antialiasing = TextServer.FONT_ANTIALIASING_MSDF   # ※要検証: アウト線崩れ(#109757)時は NONE に戻す
f.generate_mipmaps = true
f.msdf_pixel_range = 8; f.msdf_size = 48
# フォント自体に必要グリフがあればOS置換は起きない。SystemFontを使い続ける場合は allow_system_fallback=false 必須(macOS)
```

### D. 統一パレット（新規 `assets/themes/palette.gd`）
sky/sea blues, foam, wood `#8a5428`, parchment `#f3e8cd`, dark `#12283f`, gold `#e1bd72/#ffe7a8`, gauge red `#d94b4b`/green `#3cbf78`/amber `#e0a02e`, bone text `#fff1c7`。全 hex をこの const に集約。

### E. Juice 基盤
- `autoload/juicer.gd`: 専用 `ShakeLayer(CanvasLayer)` + `add_trauma(amount)`（FastNoiseLite、720p で上限〜10px）。
- `gauge_bar.gd`: 表示値の指数補間（`1 - exp(-k*delta)`）＋ゴースト追従バー ＋ダメージ点滅 ＋危険域グロー。
- `screen_base.gd make_button`: `pivot_offset=size*0.5`、hover 1.05 / press 0.92 / 離す時 `TRANS_BACK`。**必ず前の tween を `kill()`**。

## 3. gl_compatibility の“罠”（必ず守る）

| 罠 | 対策 |
|---|---|
| **2D HDR / Bloom は存在しない**（出力 0..1 でクランプ） | グローは**加算合成**か `BackBufferCopy`(1サンプル) で擬似 |
| **GPUParticles2D は macOS gl_compat で不安定**（transform-feedback） | **CPUParticles2D** を使う |
| **StyleBoxTexture に shadow が無い**（現状は shadow_size に依存） | 影を**PNG の透明マージンに焼き込む** or `StyleBoxFlat` を下に重ねる |
| **既定 texture_filter が NEAREST**（既に 0） | 装飾テクスチャには `LINEAR_WITH_MIPMAPS` を個別指定、さもなくば崩れる |
| **macOS がフォントを勝手に Hiragino 置換** | グリフを含む FontFile を使う / SystemFont なら `allow_system_fallback=false` |
| **`_draw` 内で毎フレーム `StyleBoxFlat.new()`**（badge/track/fill） | `_ready` でキャッシュ |
| **shader uniform は `tween_property` で動かせない** | `tween_method` + `material.set_shader_parameter()` |
| **`set_trans/set_ease` は“後続の”Tweenerにしか効かない** | 各 `tween_property` の前に設定 |
| **Control は左上基準で scale** | `pivot_offset = size*0.5`（サイズ判明後） |

## 4. フェーズロードマップ

- **Phase 0（基盤・即効）**: A パイプライン + D パレット + E(Juice) + `project.godot` snap。コードのみ。リスク最小、全画面に効く。
- **Phase 1（テーマ/フォント）**: B 9スライス化 + C フォント。**ウィンドウスキン PNG を“コードで procedural 生成”**してアート待ちを解消（EditorScript で `Image`→`save_png`）。フォントはフォールバック含め即適用。
- **Phase 2（看板画面）**: 水中ファイト全面（下記）。
- **Phase 3（全画面展開）**: 水上キャスト新設 / 調理・図鑑 / シェル（タイトル・港）。
- **Phase 4（仕上げ）**: コンプライアンス台帳(`docs/08`)、全体バランス、CRT/パレット後処理。

## 7. リファレンス画像の不確実性（正直な注記）

画像AI(`analyze_image`)が **`12,450 G`・`08:47`・灯台・`ビット!` などの具体値をハルシネーション**した。Read は CDN URL を返すのみで視覚埋め込みされないため、**モックアップの精密なピクセルレベル読み取りは未確定**。
→ 構成・パレット・“レトロピクセル+JRPG”という**方向性の分析は信頼できる**（コード+docs で裏付け）。**個別の hex・ラベル・個数は推定**。最終アート制作前に、実画像を実機で確認するか美術方向をロックすることを推奨。
