# 水中ファイトUI制作 振り返りレポート

Date: 2026-06-27

対象スレッド: `019ef71e-fc2b-7211-9079-1106833b8912`

このレポートは、水中ファイト看板画面の制作で「次回以降も続けるべき判断」と「無駄を生みやすかった判断」を残すためのもの。工程の時系列ではなく、今後の実装判断に使う。

## 結論

一番効いたのは、コードだけで見た目を作り込む方向から、画像素材を正式なUIパーツとして扱う方向へ切り替えたこと。背景、魚、ヒット演出、カード枠、HUD、アイコン、フォントをそれぞれ素材スロット化し、`reference/02_underwater_fight_mockup.png` と横並びで評価するようにしたことで、画面全体の品質は大きく上がった。

一方で、最も無駄になりやすかったのは、素材品質が足りない状態で明るさ、泡、線、余白、数px単位の位置を繰り返し調整することだった。素材の完成度が低いと、微調整を重ねても理想の見た目には届きにくい。次回以降は、まず良い素材候補を用意し、既存スロットへ載せた全画面比較で勝てるかを判断する。

## 良かったアクション

### 参照画像との横並び比較を標準化した

`./tools/fight_visual_qa.sh` で `/tmp/tsuri_fight_compare.png`、`/tmp/tsuri_full_static_compare.png`、`/tmp/tsuri_sidebar_static_compare.png`、`/tmp/tsuri_hud_static_compare.png` などを毎回更新するようにしたのは有効だった。

理由:

- 画面単体で見ると良く見える変更でも、参照と並べると密度、余白、色、魚の存在感、UI枠の弱さがすぐ見える。
- 古いキャプチャや一部だけのスクショを見て判断する事故を減らせる。
- 「この変更は全画面で勝っているか」という判断に戻せる。

今後も、素材候補を採用する前に必ず全画面比較を見る。

### 「P1破綻なし」と「理想品質との差分」を分けた

背景、魚位置、HUDキーサイズ、タックル行数、上部AM/時刻間隔などを、P1破綻がない限り固定したのは良かった。

理由:

- 収束条件がないと、明るさ、泡、床光、位置、文字サイズを延々と動かし続ける。
- 合格済みの項目を固定することで、次に触るべき対象が「素材品質」へ絞られた。
- 画面全体の完成度を上げる作業と、好みの微調整を分離できた。

今後は、合格済み項目を再調整する前に「P1破綻が再発しているか」を確認する。

### コード描画から素材ベースへ切り替えた

魚、背景、ヒットバッジ、右パネル、下部HUD、上部ステータス、キーキャップ、アイコンをPNG素材として扱う方針は正しかった。

理由:

- 参照画像レベルの質感、紙面、金属縁、魚の鱗、ヒレ、陰影は、コードの図形描画だけでは到達しにくい。
- 素材スロット化しておくと、本番素材が来たときに同じ座標へ差し替えられる。
- 実装側はレイアウト、状態、テキスト、ゲージを担当し、絵の質感は素材側へ逃がせる。

次回以降も「コードで全部描く」のではなく、完成度が必要な部位は画像素材として作る。

### 素材候補をいきなり採用せず、接触比較した

クロダイ素材では `tools/build_fish_asset_contact_sheet.py` を使い、候補ソース、処理後フレーム、カードポートレート、参照魚を並べて確認した。この進め方は有効だった。

理由:

- 生成画像単体では良く見えても、処理後に鱗やヒレが落ちることがある。
- 全画面に載せると、口元、頭部、白飛び、ハロ、輪郭の癖が見える。
- 「候補を作ったから採用」ではなく、「現行を明確に上回ったら採用」にできる。

魚、カード、HUD素材も同じで、候補は必ず現行との比較ボードを作る。

### 不採用理由を残した

DotGothic16の全面置換、暗すぎる魚候補、4行/6行のタックル案など、不採用理由を `design-qa.md`（現 `docs/qa/underwater_fight_qa.md`）に残したのは良かった。

理由:

- 同じ案を再試行する無駄を減らせる。
- 「なぜ今の値なのか」が後から分かる。
- 好みではなく、全画面比較で弱かった理由に戻れる。

今後も不採用にした候補は、短くても理由を残す。

### 先に公式値を固定した

右下タックルカードの「5行・13px・104pxアイコン予約幅・118x86pxロッド/リール切り抜き」のように、採用値を明文化したのは有効だった。

理由:

- 4行は薄い、6行は小さい、というループを止められた。
- 後続の素材改善でレイアウトを動かさずに済む。
- 比較対象が固定されるため、素材品質だけを評価できる。

次回も、迷いやすいUI値は比較後に正式採用値として固定する。

## 良くなかったアクション

### 仮素材を完成方向の土台として見てしまった

初期の生成PNGやコード生成風の素材は、素材ベース化の技術検証としては意味があったが、理想品質の土台としては弱かった。

問題:

- 魚のシルエット、鱗、ヒレ、目、陰影が参照に届かない。
- 背景の密度や奥行きが足りない。
- 金色の格子状UIや黒い機械箱感が、JRPG情報カードではなくデバッグUIに見える。

次回は、技術検証素材と本番候補素材を明確に分ける。技術検証が通っただけで「理想に近づいている」と判断しない。

### 微調整ループに入りやすかった

背景の中央明度、泡、床光、魚位置、HUDキーサイズ、AM間隔などを細かく触り続ける場面があった。

問題:

- 素材の根本品質が足りないと、数pxや明度の調整だけでは参照品質に近づかない。
- 一部だけ改善しても、全画面では別の弱さが目立つ。
- 合格ラインがないと、毎回「まだ少し違う」に戻ってしまう。

次回は、調整前に「これはP1破綻か、P2/P3の理想差分か」を決める。P2/P3なら、微調整ではなく素材フェーズへ送る。

### フォントを画面全体で一括置換しようとした

DotGothic16のような全面置換は、レトロ感は出ても、数値、本文、カード情報の重さが崩れた。

問題:

- 見出しには合っても本文には細すぎる。
- 全テキストを同じフォント思想で扱うと、情報カードの階層が弱くなる。
- 参照の印字感に近づくどころか、画面全体の読みやすさが落ちる。

次回は、フォントを一括で変えない。見出し、数値、本文、単位、注釈ごとに役割を分ける。必要なら文字素材化も検討する。

### パーツ単体の見た目だけで判断しそうになった

魚やHUD素材は、単体画像では良く見えても、全画面に載せると主張が強すぎる、文字が沈む、他パーツと合わないことがある。

問題:

- 生成素材単体の完成度と、画面内での完成度は別。
- カード素材が濃すぎると、情報表示ではなく黒い箱になる。
- 魚素材が強すぎると、頭部や口元だけが目立つ。

次回は、素材単体評価の後に必ず実画面比較を挟む。最終判断は全画面で行う。

### 古いキャプチャを判断材料にしそうになった

ヘッドレスGodot環境では実ランタイムキャプチャが取れない場合があり、古い `/tmp/tsuri_fishing_fight.png` が残るリスクがあった。

問題:

- 古い画像を見て、現在の変更を評価してしまう。
- 最新の静的合成とランタイムキャプチャが混ざると判断が歪む。

対策として、`tools/fight_visual_qa.sh` と比較ビルダーが最新の静的合成を正式なfallbackとして使うようになった。今後もQA画像のソース表示を見る。

## 今後の実装ルール

### 1. まず参照と現行の差分を分類する

差分は次の3種類に分ける。

- P1: 黒帯、マスク境界、残像、文字衝突、読めないゲージなど、今すぐ直す破綻
- P2: 全画面品質を明確に上げる素材差分
- P3: 好み、微細な密度、数px、微妙な明るさ

P1だけ即修正する。P2は素材フェーズとして扱う。P3だけを理由に作業しない。

### 2. 変更対象は1フェーズに絞る

背景、魚、HUD、右パネル、上部ステータス、フォントを同時に触ると、何が効いたか分からなくなる。

次回は次のように分ける。

- 魚アートフェーズ: `tools/source_assets/kurodai_final_art_source.png` を差し替える
- HUD素材フェーズ: `tools/source_assets/fight_hud_material_source.png` を差し替える
- 右パネル素材フェーズ: `tools/source_assets/sidebar_frame_material_source.png` を差し替える
- 上部ステータス素材フェーズ: `tools/source_assets/top_status_material_source.png` を差し替える
- フォント/アイコンフェーズ: 文字種別とアイコン素材だけを見る

### 3. 候補は現行より明確に勝つ場合だけ採用する

候補作成後は次を確認する。

- 現行より密度、素材感、主役感が上がっているか
- 文字、ゲージ、キー、魚、糸、ルアーが読めるか
- 参照画像に近づいたか
- 別のP1破綻を作っていないか

少し違うだけ、または一部だけ良い場合は採用しない。

### 4. 合格した値を再び動かさない

現時点の固定値:

- 背景はP1破綻がない限りfreeze
- 魚位置、魚サイズ、接地影はfreeze
- 右下タックルカードは5行、13px、104pxアイコンレーン、118x86pxロッド/リール切り抜き
- HUDキーサイズはA/B 27x24px、L/R 46x24px、+/- 24px前後
- 上部AM/時刻間隔、右パネル比率、HUD高はfreeze

次に触る場合は、固定値を動かすのではなく、素材の質を上げる。

### 5. ドキュメントは判断ログとして使う

`docs/qa/underwater_fight_qa.md`（旧 `design-qa.md`）は採用値と判断ログ、`docs/11_underwater_fight_showcase.md` は実装仕様、`docs/12_underwater_fight_production_asset_brief.md` は次素材の条件として使い分ける。

今後、候補を採用または却下したら、最低限次を残す。

- 何を変えたか
- 何を変えていないか
- どの比較画像で判断したか
- 採用または却下した理由
- 次に同じループへ戻らないための固定条件

## 次回以降の推奨手順

1. `./tools/fight_visual_qa.sh` を実行して最新比較を作る。
2. `/tmp/tsuri_fight_compare.png` と `/tmp/tsuri_full_static_compare.png` を見る。
3. P1破綻がないかだけ先に確認する。
4. P1がなければ、背景や配置ではなく、素材フェーズを1つ選ぶ。
5. 候補素材を作る。
6. 既存スロットへ仮適用する。
7. 現行/候補の比較ボードを作る。
8. 全画面で明確に勝つ場合だけ採用する。
9. `py_compile`、`git diff --check`、`./tools/validate_project.sh` を通す。
10. 判断理由を `docs/qa/underwater_fight_qa.md` または該当docsへ残す。

## 次にやるなら

最も効果が出やすい順番は次の通り。

1. 右パネルまたはHUDの本番カード素材をさらに強い完成素材へ差し替える。
2. 現行クロダイを明確に上回る魚アート候補がある場合だけ、魚ソースを再選定する。
3. UI専用フォント/文字素材化を検討する。
4. 最後に、全画面で残る小さなP2/P3差分をまとめて見る。

避けるべき順番は、背景明度、泡、魚位置、HUDキーサイズ、タックル行数、AM間隔を再び個別に触ること。そこはv1収束で固定済みであり、今後の品質差は主に素材の完成度で埋める。
