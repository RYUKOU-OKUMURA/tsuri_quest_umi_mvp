# 技術設計

## 1. アーキテクチャ

```text
Main（画面ルーター）
├─ TitleScreen
├─ HarborScreen
├─ FishingSpotSelectScreen
├─ FishingScreen
│   ├─ FishingSimulator
│   ├─ UnderwaterView
│   └─ SurfaceCastView
├─ CookingScreen
├─ MarketScreen
├─ ShopScreen
└─ StatusScreen

Autoload
├─ GameData        静的マスターデータ
└─ PlayerProgress  進行・所持品・セーブ
```

## 2. 責務

### `src/main.gd`

- 現在画面の生成と破棄
- 画面遷移シグナルの受信
- 終了時セーブ

### `src/autoload/game_data.gd`

- 魚、釣り場ポイント、料理、竿、船のマスターデータ
- 通常魚の抽選
- 釣り場ポイントごとの出現weight補正
- 釣り場の発見Lvと船ランクによる出航可否判定
- 魚サイズの抽選
- 料理経験値計算

### `src/autoload/player_progress.gd`

- レベル、経験値、所持金
- クーラーボックス、図鑑、料理記録
- 装備購入・切り替え
- 船購入と所持船による出航可否判定
- 食事効果の保留と消費
- JSONセーブ／ロード

### `src/core/fishing_simulator.gd`

- 釣り状態機械
- 魚行動抽選
- テンション、体力、距離、水深の計算
- 捕獲／逃走判定

### `src/ui/components/underwater_view.gd`

- シミュレーターの値を視覚化
- 水中背景、魚、糸、エサの描画
- ゲームルールを持たない

### `src/ui/fishing_spot_select_screen.gd`

- 解放済み釣り場ポイントの表示
- 未解放ポイントのロック表示
- 選択した `spot_id` を釣り画面へ渡す
- 地図アートではなく、初期実装ではカード式UIを優先する

## 3. 釣り状態機械

```text
READY
  ↓ cast
CASTING
  ↓
WAITING
  ↓
APPROACH
  ↓
BITE
  ├─ hook → FIGHT
  └─ timeout → ESCAPED
FIGHT
  ├─ success → CAUGHT
  └─ line break/slack → ESCAPED
```

## 4. セーブ形式

保存先：`user://tsuri_quest_save.json`

主要フィールド：

```json
{
  "version": 1,
  "level": 1,
  "exp": 0,
  "money": 500,
  "inventory": {"aji": 2},
  "caught_counts": {"aji": 3},
  "spot_caught_counts": {"harbor_pier": {"aji": 2}},
  "best_sizes": {"aji": 31.4},
  "eaten_recipes": {"aji:salt_grill": 1},
  "owned_rods": ["starter"],
  "equipped_rod_id": "starter",
  "owned_boats": [],
  "pending_buff": {},
  "play_seconds": 0.0
}
```

## 5. 拡張方針

正式開発では、マスターデータをカスタムResourceへ移す。

```text
FishData.tres
RecipeData.tres
RodData.tres
FishingSpotData.tres
```

MVPでは調整箇所を見つけやすくするため、辞書に集約している。

## 6. エラー方針

- 存在しないIDは空辞書または失敗結果を返す。
- UI側は失敗結果のメッセージを表示する。
- セーブの解析に失敗した場合は初期値を保持する。
- ゲームプレイ中に例外相当の状態が発生しても、港へ戻れる設計を維持する。
