# 19. 画面UI制作プレイブック（標準指示書）

Date: 2026-07-02

この文書は、今後の画面UIデザイン実装を担当するエージェント/作業者への**標準指示書**である。
`docs/13_underwater_fight_process_lessons.md` と `docs/17_fishing_spot_map_process_lessons.md` の2本の振り返りで**両方に収束した教訓**をプロセスルールへ昇格させ、さらに両レポートに欠けていた「画面横断の一貫性」を追加したもの。

- 位置づけ: `docs/10_UIクオリティ向上マスタープラン.md` が「何を目指すか」、本書が「毎回どう進めるか」。矛盾した場合は本書を優先し、docs/10 側を更新する。
- 各画面固有の採用値・freeze値は本書に書かない。画面別のQAドキュメント（`docs/qa/<screen>_qa.md`。書式と運用は `docs/qa/README.md`）へ記録する。
- 本書のルールを変更したいときは、変更理由を本書の末尾「改訂履歴」に残してから変える。黙って逸脱しない。

## 0. 最重要原則（迷ったらここへ戻る）

1. **構成 → 主導線 → 情報階層 → 文字の収まり → 素材の質感 → 細部の演出**。この順番を逆にしない。構成がズレたまま細部を磨くのは、労力の割に完成イメージへ近づかない（17番の8セル台帳の教訓）。
2. **質感はPNG素材、テキスト・状態・ゲージ値はGodot runtime描画**。日本語テキストを素材に焼き込まない。
3. **微調整より素材差し替え**。数px・明度の調整を3回繰り返しても参照に近づかないなら、それは調整不足ではなく素材品質不足。素材フェーズへ送る（13番の教訓）。
4. **判断は必ず全画面比較で行う**。パーツ単体で良く見えても、全画面で勝っていなければ不採用。
5. **合格した値はfreeze**。P1破綻が再発しない限り再調整しない。
6. **共通キットにある部品を画面ごとに再生成しない**（本書 §3。これが過去2画面に欠けていた視点）。

## 0.1 差分トリアージ（P1/P2/P3）

参照画像と現状の差分は、作業前に必ず3種類へ分類する。

| 区分 | 定義 | 扱い |
|---|---|---|
| P1 | 破綻。黒帯、マスク境界、残像、文字衝突・見切れ、読めないゲージ、アイコンと数値の重なり、素材未表示 | 即修正。他作業より優先 |
| P2 | 全画面品質を明確に上げる素材・構成差分 | 素材フェーズ/構成フェーズとして1つずつ実施 |
| P3 | 好み、微細な密度、数px、微妙な明るさ | **P3だけを理由に作業しない** |

## 1. 着手前フェーズ：参照画像の分解ドキュメント（必須）

新しい画面に着手する前に、対象の完成イメージ（`reference/` 内の正式画像）を分解し、短いドキュメント（画面別spec docへの追記でよい）にする。**これを書かずに実装を始めない。**

分解の観点:

- **1状態の画面か、複数状態の合成モックか**: 参照画像は複数の画面状態を1枚に合成していることがある（例: `03_cooking_levelup_mockup.png` は「料理を選ぶ」「食べる」「効果獲得」「レベルアップ」の4状態の合成）。合成モックを1画面に全部詰め込むと、情報の重複と過密が必ず起きる。どの状態がどのフェーズ/オーバーレイに対応するかを先に決める。
- **存在する領域**: 上部/右/下部/中央に何があるか。領域ごとの役割。
- **存在しない領域**: 参照に無いのに作りたくなる要素を明記する（例: 17番の下部8セル台帳）。作り込むほど完成イメージから離れる要素を先に潰す。
- **主操作**: この画面でユーザーが一番押すもの。1つに絞る。
- **補助操作**: 副ボタン・メニュー類。
- **データ表示**: runtime で変わる値（名前、数値、達成度、状態）。
- **素材が担う部分 / Godotが描く部分** の分担表。
- **共通キットで賄える部分 / この画面専用に生成する部分** の分担表（§3）。

参照画像が途中で更新されたら、差し替え時点でこの分解をやり直し、既存実装との矛盾を洗い出す。古い構成を磨き続けない。

## 2. 品質基準（完了条件の定義）

「80点」「90点」を主観にしないため、以下をチェックリストとして使う。

### 2.1 v1合格（80点 = 本番寄り画面）の完了条件

すべて満たしたら v1 として収束を宣言し、freeze する。

- [ ] 分解ドキュメントの「存在する領域」がすべて実装され、「存在しない領域」が残っていない
- [ ] 1280x720 の実スクショで、文字の見切れ・省略（`アンコ…` のような ellipsis 表示）・重なりがゼロ。全Labelに `clip_text` と `text_overrun_behavior` が設定済みで、かつ通常データで省略が発動しない幅設計になっている
- [ ] **同一情報が同一画面に2箇所表示されていない**（例: Lv・所持金のヘッダーとフッターの重複）。複数状態合成モック由来の詰め込みは§1の状態分解へ戻す
- [ ] 主操作ボタンが画面内で一番目立つ（ぼかしテストで判別できる）
- [ ] 質感が必要な部位がPNG素材化済みで、フォールバック表示になっていない
- [ ] 参照画像との横並び比較で、領域構成・読み順が一致している
- [ ] 上部・右・下部が同じ世界観に見える（§3の共通キット使用）
- [ ] 装飾階層ルール（§4.4）に違反していない（行レベルの金縁フレーム・行端飾りが無い）
- [ ] 数値表記ルール（§4.3）に準拠している
- [ ] `tools/<screen>_visual_qa.sh` が存在し、参照との横並び比較画像を再生成できる
- [ ] smoke test が**現行UX契約**を検証している（古い構成を正とするテストが残っていない）
- [ ] `./tools/validate_project.sh` と該当画面のsmokeが通る
- [ ] 画面別QAドキュメントに採用値・freeze値・不採用理由を記録した

### 2.2 v2（90点 = 看板品質）の完了条件

v1収束後、別フェーズとして実施する。v1作業と混ぜない。

- [ ] 画面専用の一点物素材（魚アート、釣り場サムネイル等）が「現行に全画面比較で明確に勝つ」候補へ差し替わっている
- [ ] 見出し・数値・本文・ボタンでタイポグラフィの役割分担ができている（§4.2）
- [ ] 主要な状態変化に演出（点滅・スライドイン・バースト等）が付いている
- [ ] スクショを縮小してサムネイルサイズで見ても画面の魅力が伝わる（1枚絵テスト）
- [ ] グレースケール化しても情報階層が読める（明度設計テスト）

## 3. 共通アセットキット規約（画面横断の一貫性）

過去の画面は画面ごとに枠素材をゼロから生成した（`generate_cooking_showcase_assets.py` → `cooking/`、`generate_fishing_spot_map_assets.py` → `fishing_spots/` のように、生成スクリプトも素材フォルダも画面別）。画面数が増えると金縁の太さ・紙の色味・濃紺の彩度が画面ごとにズレて「同じゲームに見えない」問題が出る。実際、調理場・魚図鑑・釣り場マップの3画面で上部バー・戻るボタン・帯の様式がすべて異なっている。以後は次を守る。

**現状（2026-07-02 改訂2時点）**: `assets/showcase/common/` に共通キットv0を設置済み。魚図鑑の `harbor/`・`fishing_spots/`・`cooking/` 直接参照は解消済みで、魚素材はUI部品ではなく `assets/showcase/fish/` のドメイン共有素材として扱う。**共有したくなった部品は `common/` へ昇格させてから両画面で参照する**こと。他画面のフォルダを直接参照しない。

### 3.1 キットの構成

`assets/showcase/common/` に、全画面共通のUI部品を置く。v0ではステータスバー、汎用ボタン、紙/カード枠、行枠、汎用ステータス/導線アイコンを置く。背景、魚、料理、釣り場サムネは入れない。

- フレーム: 金縁+濃紺外枠の9-slice（角・辺・面）、パネル用と小カード用
- 紙面: 羊皮紙テクスチャ（明・中の2トーン）
- 帯: 濃紺ヘッダー帯、深緑ヘッダー帯
- ゲージ: セル型ゲージの枠・空セル・グラデーション面（テンション用 緑→黄→赤 / 体力用 緑→シアン）
- キーキャップ: Space / Shift / E / Enter / +/- 等の実キー表記セット
- 汎用アイコン: コイン、時計、天候、風、魚シルエット等

### 3.2 運用ルール

- **新しい画面のパネル・枠・帯・ゲージ・キーキャップは、共通キットの組み合わせのみで構成する。**
- **上部ステータスクラスタ（Lv / 所持金 / 装備竿など）は共通コンポーネントにする。** 汎用画面は `src/ui/components/player_status_bar.gd` を使う。表示順の文法も固定する: **Lv → 装備 → 所持金** の順、所持金は常に右端。水中ファイト専用HUDは `FightStatusBar` を維持する。
- 「港へ戻る」系の戻るボタンは `ScreenBase.make_return_button()` を使う。
- 「港へ戻る」導線は画面右下に置く（規約。詳細と経緯は `docs/28_harbor_return_placement_unification.md`）。釣行中の中断確認など、別UX契約の導線は例外としてQAログに明記する。
- 画面専用に新規生成してよいのは「キットに無い一点物」だけ（魚アート、背景、料理イラスト、木製看板のような画面固有シンボル）。
- キット部品の見た目を変えたくなったら、その画面だけ変えるのではなく**キット自体を更新し、全画面へ波及させる**。波及確認は各画面の比較スクショで行う。
- 基準画面は水中ファイト（`reference/02_underwater_fight_mockup.png`）。キット初期構築は水中ファイトの採用素材からの切り出しを優先する。
- 一点物の新規生成には `docs/12_underwater_fight_production_asset_brief.md` の形式で発注仕様（サイズ、スタイル、禁止事項、比較基準）を書いてから生成する。
- `src/ui` からの素材所有違反は `tools/audit_showcase_asset_refs.py` で検出する。例外が必要な場合は同ツール内の allowlist に理由コメントを残す。

### 3.3 生成素材の統一処理

AI生成素材は出力ごとに色・密度がブレる。採用前に必ず統一処理を通す。

- **共通パレットへの寄せ**: §4.1 のパレットへ量子化または色相寄せする処理を、既存の PIL パイプライン（`tools/generate_underwater_ui_frame_assets.py` 系）に組み込んで適用する。
- **解像度規約**: 表示サイズの等倍または2倍で生成し、縮小して使う。画面内でピクセル密度が混ざらないようにする。
- **なじませレイヤー**: 背景に載せる一点物（魚等）は、環境色ティント（背景色を10–20%乗算）、リムライト、前景パーティクルの3点セットで統合する。ステッカー感が残る素材は未完成とみなす。
- テキスト焼き込み禁止、透過はクロマキー方式（キー色は生成枠から実測サンプリング。`#ff00ff` 固定を仮定しない）。

### 3.4 一点物素材の供給経路（プロシージャル生成の適用範囲）

素材の作り方には2経路あり、対象で使い分ける。**混同すると品質が頭打ちになる**（2026-07-05 内部調査より: `generate_cooking_showcase_assets.py` のPIL描画による王冠・月桂樹・バースト演出は、参照画像の質感に3スライス連続で未到達だった）。

| 経路 | 適用対象 | 適用禁止 |
|---|---|---|
| プロシージャル生成（PIL等のコード描画） | 幾何部品: 枠・帯・9-slice・ゲージ枠・キーキャップ・セパレータ・単色チップ | 有機的・絵画的な一点物 |
| AI画像生成（発注仕様→生成→§3.3統一処理） | 一点物: 背景イラスト、キャラクター、料理・魚アート、王冠/月桂樹/メダル等の装飾モチーフ、大型演出（バースト・紙吹雪） | — |

- 一点物をプロシージャル描画で作って「候補」として採用しない。参照級の質感が必要な部位は、docs/12形式の発注仕様を書き、画像生成で候補を作る。生成手段が使えない環境では、その部位を**素材待ちP2**としてQAドキュメントに起票し、プロシージャル代替で「前進」扱いしない。
- 参照画像からの切り抜きは分解作業の暫定素材としては許容するが、正式素材ではない（AGENTS.md「reference/は最終アートではない」）。切り抜きのまま残っている部位はQAドキュメントの残ギャップに明記する。

## 4. スタイルガイド（デザイントークン）

### 4.1 パレット

**正本は `src/ui/palette.gd`**（`GOLD_BRIGHT #ffe7a8`、`PARCHMENT #f3e8cd`、`DARK_PANEL #13283f`、`TEXT_BONE #fff1c7` 等が定義済み）。運用ルール:

- **runtime描画の色は必ず `Palette.*` を参照する。新規のハードコード hex を画面スクリプトに書かない。** 現状は各画面に大量のハードコード色が散在している（例: `cooking_screen.gd` の `#25170e`、`fish_book_screen.gd` の `#ead4a4` 等）。触った画面から順に palette.gd のトークンへ置換する。
- 足りない色は palette.gd にトークンを追加してから使う（用途名で命名。値の直書き禁止）。
- 生成PNG素材も同じパレットへ寄せる（§3.3）。素材とruntime描画で「同じ紙色のはずが違う」状態を作らない。

参照画像からの抽出仮値（キット構築時に面平均で再抽出し、palette.gd と統合して確定すること）:

| トークン | 仮採用値 | 用途 |
|---|---|---|
| paper-light | `#ecddbf` | カード紙面（明） |
| paper-dark | `#a39784` | 紙面の影・縁汚し |
| navy-deep | `#062543` | 濃紺帯・数値ボックス |
| green-header | `#044335` | 深緑ヘッダー帯 |
| gold-ivory | `#f8ebd0` | 金縁ハイライト・帯上の見出し文字 |
| gauge-green | `#26d538` | ゲージ安全域 |
| gauge-orange | `#f08c1c` | ゲージ警告域 |

（危険域の赤、水中の基調ブルー2種、HUD暗色帯は確定時に追加する）

**レアリティ表記の規格**（`src/ui/rarity_styles.gd` と `src/ui/palette.gd` を正とする）:

- チップ色: コモン=青系 / アンコモン=緑系 / レア=紫系 / ぬし=金系（参照 `07_fish_book_mockup.png` に準拠）
- チップは**文言が省略されない最小幅**を規格化する。`アンコ…` のような省略表示はP1。幅が確保できないレイアウトなら、チップ幅を文言に合わせて設計し直す（文言を削らない）。
- レアリティ文言・色は `RarityStyles` 経由で参照し、図鑑・釣り中カード・調理場で共有する。

### 4.2 タイポグラフィの役割分担

フォントの一括置換は禁止（13番でDotGothic16全面置換が失敗済み）。役割ごとに分ける。

**現行の正式フォントは LINE Seed JP**（`src/ui/game_fonts.gd` が Rg/Bd/Eb の3ウェイトを読む。M PLUS 1p は過去採用で、現行コードでは使っていない。旧 `fight_fonts.gd` は2026-07-05に削除済み）。

| 役割 | 指定 |
|---|---|
| 大数値（時刻・金額・水深・cm） | LINE Seed JP ExtraBold。画面内で最も太く大きく |
| 見出し（帯上のカードタイトル） | ExtraBold + アイボリー文字 + 1px落ち影。わずかに letter-spacing |
| 本文・説明文 | LINE Seed JP Regular/Bold。小さい日本語は装飾より可読性優先 |
| ボタンラベル | Bold。アイコンとテキストは自前配置（Button標準の icon+text 任せにしない） |
| 単位・注釈 | Regular 小サイズ。数値より1段細く |

**アンチエイリアス方針（2026-07-05 確定）**: 全UIテキストは `game_fonts.gd`（AA有効: FONT_ANTIALIASING_GRAY / HINTING_LIGHT / subpixel 1/4）に統一する。`fight_fonts.gd`（AA無効）は削除済み。決定理由: ①実利用が水中ヒット演出のテキストフォールバック1箇所のみで、全画面の実態は既にAA有効だった、②当該箇所はPNGバッジ素材（`hit_badge_full.png`）があればテキスト描画に到達しない構造で、AA無効の質感が本番画面に出ることがなかった、③LINE Seed JPはドットフォントではなくAAオフでは中途半端なジャギーになる。ドット質感の演出が必要になった場合はフォントAAではなくPNG素材（焼き込みテキスト禁止ルールに従い図像のみ）で表現する。全画面に効くフォント変更は、1画面で試して比較ボードを作ってから、キット方針として本書に反映する。

### 4.3 数値表記ルール

- 金額は桁区切りカンマ必須（`12,450 G`）
- 数値は右揃え・等幅（tabular figures 相当の桁揃え）
- アイコン領域と数値領域を分離し、重なりを禁止（P1扱い）
- 単位（G / m / cm / 匹）は数値より小さく

### 4.4 装飾階層ルール（過密化の防止）

参照モックの高級感は「枠が少ない」ことで成立している。金縁フレームはパネル階層にだけ使い、内側は静かにする。

| 階層 | 装飾 |
|---|---|
| 画面外周・大パネル | 金縁+濃紺の本フレーム（9-slice）。飾り金具・角プレート可 |
| パネル内のカード（料理カード、魚カード等） | 細い単線ボーダー+紙面。選択中のみ金ハイライト |
| 行・リスト項目 | 罫線・背景色の交互差・チップのみ。**行ごとの金縁フレームと行端の飾り金具は禁止** |

- 1画面内の「枠の中の枠」は最大3段（画面→パネル→カード）。4段目が必要になったら情報設計を疑う。
- 状態表示（初回済・残回数など）は浮いたテキストにせず、チップまたは行内の右端スロットに収める。

### 4.5 背景と一覧パネルの関係

一覧・図鑑・メニュー系画面では、背景イラストが情報と喧嘩しないようにする。方式は2つのどちらかを選ぶ:

1. **密閉型**（図鑑向き）: 参照 `07_fish_book_mockup.png` のように画面全体を木枠台帳で囲い、背景は縁にしか見せない。
2. **減光型**: 背景の上に暗色スクリム（20〜40%）またはブラーを敷いてからパネルを載せる。

背景をフルの明度・彩度のままパネル背後に敷くのは禁止（現状の魚図鑑がこの状態）。ゲームプレイ画面（水中ファイト・マップ）は背景が主役なのでこのルールの対象外。

### 4.6 基盤レイアウト原則（ゲームプレイ画面。2026-07-07ユーザー決定）

釣行画面ファミリー（水上READY〜水中ファイト）の基盤デザイン。参照 = `reference/13_fishing_ready_danger_mockup.png`（READY）/ `reference/14_underwater_fight_simple_mockup.png`（FIGHT）。新規実装・upliftはこの原則を既存freeze値より優先する（既存freezeの上書きは各画面QAドキュメントに判断ログを残すこと）。

1. **シーン優先**: 中央の実プレイシーン（水面・水中）に画面の7割前後を明け渡す。UIクロームは上部1列＋下部1本のスリムバーに集約する
2. **下段スリムバー**: 状態ごとに中身が切り替わる1本のバー。READY=餌魚セレクタ＋大型主操作、FIGHT=テンション／主操作2つ／魚の体力。その状態で意味を持たない情報（READY中のゲージ等）は表示しない
3. **主操作は1つ、大きく**: その状態でプレイヤーがすべき操作（投げる・アワセ・巻く/糸を出す）を画面の主役サイズにし、キーチップを添える
4. **サイドバーよりフローティングカード**: 常設の縦サイドバーは置かず、必要な情報（魚名・推定サイズ・行動ヒント等）はシーン右上の小型カードに集約する。カードは1枚まで
5. **色・質感・タイポは §4.1〜4.4 のまま**: 本原則はレイアウト規律であり、新しいアートスタイルではない。共通キットを再配置して使う

メニュー系・一覧系画面（港・図鑑・市場・生簀など）は本原則の対象外（§4.5の密閉型/減光型に従う）。

## 5. 実装中ルール

1. **1フェーズ1変更**。背景・主役素材・HUD・右パネル・上部・フォントを同時に触らない。何が効いたか分からなくなる。
2. **候補は接触比較→全画面比較の2段階**。候補ソース／処理後／現行／参照を並べた contact sheet（`tools/build_fish_asset_contact_sheet.py` 方式）で確認後、既存スロットへ仮適用して全画面比較ボードを作る。
3. **採用条件は「現行に明確に勝つ」**。少し違うだけ・一部だけ良い候補は不採用。不採用理由を必ず残す。
4. **技術検証素材と本番候補素材を区別する**。仮素材でパイプラインが通ったことを「品質が近づいた」と誤認しない。
5. **freeze値の尊重**。画面別QAドキュメントに記録されたfreeze値（位置、サイズ、行数、間隔）は、P1破綻の再発時以外は動かさない。次の品質向上は値いじりではなく素材差し替えで行う。
6. **見た目の変更でもゲームフローを検証する**。UI導線は `trip_stats`、食事バフ、釣行継続などの状態に接続していることがある（17番の教訓）。該当smoke・監査・validateを必ず回す。
7. **新規PNGの読み込み確認**。`.import` 生成前の新規素材は `ResourceLoader.exists()` だけでは読めないことがある。`ImageTexture.create_from_image()` フォールバック（既存実装あり）を使い、スクショで実表示を確認する。素材が出ていない状態でレイアウトを判断しない。
8. **古いキャプチャで判断しない**。比較画像のソース表示（どのスクショ・いつの合成か）を毎回確認する。
9. **フェーズ開始時にスコープを宣言する**。P2フェーズに着手する前に、「今回動かすパラメータ」と「触らないfreeze値」を画面別QAドキュメントのスコープ宣言欄へ書き出す。宣言外の値を触りたくなったら、その場で触らず別フェーズとして起票する（水中ファイトで背景・魚・HUD・上部・フォントが同一パス内で並行して動いた反省より）。
10. **微調整の回数を数える**。同一パラメータ（位置・サイズ・明度など）の微調整は、QAドキュメントの微調整カウンタへ記録する。3回で改善しなければ原則3（素材差し替え）を適用し、そのパラメータをfreezeして素材フェーズへ送る。数えずにループしない（旧ログでは魚位置6回以上・背景約30パスの反復が起きた）。
11. **採用判定はbefore勝ちと参照距離の2条件で行う**。フェーズ着手時に「参照との差分Top3」を面積×視線優先度で書き出し、原則その最上位に着手する（最上位を飛ばす場合は理由をQAドキュメントへ）。完了判定は「beforeに明確に勝つ」に加えて、**対象にした差分が縮小サムネイル比較（after/参照を並べて縮小表示）で縮んだと判別できる**こと。beforeとの微小差だけを根拠に採用を積み重ねない（2026-07-05の調理フローで「参照ほどではないが前進」の連続採用が起き、体感品質が上がらなかった反省より）。
12. **「参照未到達だが前進」は同一画面で3フェーズまで**。同じ画面・同じ系統の差分（例: 演出の迫力、装飾の豪華さ）に対して「参照ほどではないが前進した」判定を3フェーズ連続で出したら、それはアプローチ自体の限界とみなし、微調整3回ルールと同様に構成フェーズまたは素材フェーズ（§3.4のAI生成経路）へエスカレーションする。
13. **runtime描画の装飾追加は1画面につき累計3パスまで**。`StyleBoxFlat`・罫線・wash・影などの装飾を「新しいパラメータ名」で追加しても、画面単位の**装飾パス累計**として微調整カウンタに数える。累計3パスを超えて質感が参照に届かないなら、runtime装飾の積み増しをやめて素材フェーズへ送る（魚図鑑で装飾追加パスが約24回積み上がり、1パスあたりの改善が判別困難になった反省より）。

## 6. 検証（visual QA）

### 6.1 スクショQAの標準化

- 画面ごとに `tools/<screen>_visual_qa.sh` を用意する（`fight_visual_qa.sh` / `cooking_visual_qa.sh` が雛形）。固定スクショ生成＋参照との横並び比較画像の再生成を1コマンドにする。**現状、魚図鑑と釣り場マップはこのスクリプトが未整備（smoke testのみ）。両画面の次回作業時に最初に作ること。**
- **実スクショなしで完了判断しない**。コード上の size / custom_minimum_size では日本語のはみ出しは検出できない。
- スクショはQAツール経由の正確な 1280x720 キャプチャを使う。デスクトップのウィンドウキャプチャ（OSのUIが写り込む、スケールが混ざる）を判断材料にしない。
- **ランタイムキャプチャが使えない環境**（headless `SubViewport` 不可、プレビュークラッシュ等）では、静的合成比較（`tools/build_fight_*_static_compare.py` 方式）で作業を進めてよい。ただしその合否判定は**暫定**とし、QAドキュメントの「暫定判定・再検証TODO」欄へ集約する。キャプチャ復旧後に必ず実スクショで再判定する。暫定判定を根拠にfreezeを恒久化しない。
- **採用判断の根拠にした比較画像は `docs/qa/evidence/<screen>/` へ日付付きファイル名でコピーする**。`/tmp` はセッションで消えるため、`/tmp` のみに残した証拠は後から検証不能になる（水中ファイトv1パスの証拠約20枚が失われた反省より）。

### 6.2 スクショで毎回見る項目

- 文字の見切れ・省略・重なりがないか
- 余白に不自然な穴がないか
- 主ボタンが一番目立つか / 選択操作の場所が一目で分かるか
- 新規素材が実表示されているか（フォールバックになっていないか）
- 上部・右・下部が同じ世界観に見えるか
- （v2以降）グレースケール比較で明度設計が参照と揃っているか、縮小1枚絵で魅力が出ているか

### 6.3 テストの更新

構成を変えたら smoke test を**新しいUX契約**へ更新する。古い構成を正とするテストを残すと次の作業で巻き戻る。「存在しないこと」の検証（例: 撤去した要素が復活していない）も入れる。

## 7. 記録規約（判断ログ）

ドキュメントの使い分け:

- **画面別QAドキュメント**（`docs/qa/<screen>_qa.md`）: 採用値・freeze値・不採用理由・微調整カウンタ・暫定判定。**1画面1ファイル・日本語**。書式と運用は `docs/qa/README.md` のテンプレートに従う
- **画面別spec/showcase doc**: 実装仕様と分解ドキュメント
- **発注仕様doc**（docs/12方式）: 次に作る素材の条件
- **本書**: 全画面共通のプロセスとトークン。画面固有の値は書かない

QAドキュメントの品質ルール（旧 `design-qa.md` が英文・追記専用の約130箇条／5万トークン級ログとなり、次の作業者のfreeze値抽出コストが激増した反省より）:

1. **追記専用にしない**。値を更新したらfreeze表の該当行を上書きし、上書きされた（superseded）判断ログは削除する。詳細な経過を残したい場合は `docs/qa/archive/` へ移す。
2. **freeze値は表で書く**（項目/値/場所/理由）。長文の中に数値を埋めない。
3. **不採用案は専用リストに書く**（案/却下理由/判断日）。次の作業者が履歴を全読みしなくても再試行を避けられる状態を保つ。
4. **他画面の記録を混ぜない**。別画面の作業をしたら、その画面のQAファイルへ書く。

候補を採用/却下したら最低限これを残す: 何を変えたか / 何を変えていないか / どの比較画像で判断したか（＋evidenceへのコピー先） / 採用・却下理由 / 同じループへ戻らないための固定条件。

## 8. 標準手順（1画面の進め方）

```
[着手前]
1. 参照画像の分解ドキュメントを書く（§1）。存在しない要素を明記
2. 共通キットで賄う部品と、画面専用の一点物を仕分ける（§3）
3. smoke testの検証観点を新UX契約で定義する

[構成フェーズ]
4. 領域構成・主導線・情報階層を参照に合わせる（共通キット部品で組む）
5. 実スクショ＋横並び比較で構成一致を確認。文字の収まりを直す

[素材フェーズ]（1回につき1スロット）
6. 発注仕様を書く → 候補生成 → 統一処理（§3.3）
7. contact sheet比較 → 仮適用 → 全画面比較
8. 明確に勝てば採用、負ければ不採用理由を記録

[収束]
9. §2.1 のv1チェックリストを全て満たす
10. validate / smoke / 監査を通す
11. 採用値でQAドキュメントのfreeze表を上書きし、判断根拠の比較画像を docs/qa/evidence/ へコピー
12. v2（90点）作業は別フェーズとして起票する
```

## 9. アンチパターン（やらないこと）

- 分解ドキュメントなしで実装を始める
- 参照に存在しない要素を磨き続ける
- 構成がズレたまま素材・細部を磨く
- 素材品質不足を数px・明度調整で埋めようとする
- P3（好み）だけを理由に freeze 値を再調整する
- フォントの全画面一括置換
- パーツ単体の見栄えだけで採用判断する
- 共通キットにある部品を画面ごとに再生成する
- 他画面の素材フォルダを直接参照する（共有するなら `common/` へ昇格）
- 複数状態の合成モックを1画面へそのまま詰め込む
- 行・リスト項目に金縁フレームや飾り金具を付ける
- 明るい背景イラストの上へスクリムなしで情報パネルを載せる
- 同一情報を同一画面に2箇所表示する
- 古いスクショ・部分スクショで判断する
- 見た目だけの変更だと思い込んでゲームフロー検証を省く
- テストを古いUX契約のまま残す
- QAドキュメントを追記専用で肥大化させる（supersededの記録を消さない）
- freeze値を長文の中に埋める（表で書かない）
- 1つのQAファイルに複数画面の記録を混ぜる
- 採用判断の証拠画像を `/tmp` にしか残さない
- 微調整の回数を数えずに同一パラメータを触り続ける
- フェーズのスコープ宣言なしにP2作業へ着手する
- 背景・キャラ・料理・装飾モチーフ等の一点物をプロシージャル描画で作って採用する（§3.4）
- beforeとの微小勝ちだけを根拠に採用を積み重ねる（参照距離を測らない）
- 装飾に新しい名前を付けて微調整カウンタ・装飾パス累計を回避する
