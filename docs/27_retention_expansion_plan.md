## 2. 設計方針 — リテンションの環

本拡張の目的は、釣行の動機を「レベル上げ」一本から解放し、次の環を閉じること:

```
「今日は何を狙うか」──→ 釣行 ──→ 「記録更新・発見」──→ 「次の目標」──┐
        ↑ 依頼ボード(E3)         記録演出・称号(E1)      ヌシ図鑑(E2)   │
        └──────────────────────────────────────────────────────────────┘
                     環の最終目標 = サメ危険海域(E4)
```

- **E1〜E4 がコア**。この4つで環が閉じる。順序を崩さない
- E5〜E6 は環に変化を付けるスパイス、E7〜E8 はポリッシュ、E9 は横展開の仮説検証
- 難易度・ミニゲームを先にやらない。環がない状態でポリッシュしても遊ぶ理由は増えない

## 3. 依存グラフと推奨順序

### 依存グラフ

```
[E1 記録演出＋称号] ─────────────────────────┐（独立・最小）
[E2 ヌシシステム] ──┬─→ [E4 サメ危険海域]     │
                    │        ↑                │
[E3 依頼ボード] ────┼─→ [E8 ザリガニ釣り]     │
                    │                          │
[E5 時間帯] ────────┘（ヌシ出現条件を豊かにする。E2着手後いつでも）
[E6 ランダムイベント] …… E4の解放演出（ボトルメール→海図）に使えるためE4前が望ましい
[E7 難易度選択] …… 独立。ただしE1〜E4完了後（§2の方針）
[E9 川エリア] …… 全フェーズの後。カタログ差し替えで成立するかの横展開検証を兼ねる
```

### 推奨順序と規模感

| 順 | ID | 内容 | 種別 | 規模感 | 新素材 |
|---|---|---|---|---|---|
| 1 | **E1** | 記録更新演出＋称号 | 演出＋データ | 小 | 不要（既存UI流用） |
| 2 | **E0** | レベルキャップ拡張（10→30。2026-07-05決定で新設） | データ | 小 | 不要 |
| 3 | **E2** | ヌシシステム | システム＋データ | 中 | 小（金枠・ヌシ演出程度） |
| 4 | **E3** | 依頼ボード | 新画面＋システム | 大 | 要（画面素材一式） |
| 5 | **E6** | 釣行中ランダムイベント | システム＋演出 | 中 | 小 |
| 6 | **E4** | サメ危険海域 | データ＋システム | 中 | 要（サメ2〜3種×2点セット＋海域サムネ） |
| 7 | **E5** | 時間帯（朝まずめ・夜釣り） | データ＋演出 | 中 | 中（時間帯別の背景・空气感） |
| 8 | **E7** | 難易度選択 | システム | 小 | 小（タイトル画面の選択UI） |
| 9 | **E8** | ザリガニ釣りミニゲーム | システム＋データ | 中 | 要（ザリガニ素材＋淡水演出） |
| 10 | **E9** | 川エリア（横展開検証） | データ＋素材 | 特大 | 特大（川魚・背景・釣り場一式） |

E0 は E4（サメ海域=Lv30）・E5（夜釣り=Lv15）の前提。現行コードの `MAX_LEVEL` は 10 のため、キャップを30へ拡張し Lv11 以降はステータス成長を減衰させて主に解放ゲートとして使う（詳細は docs/28 §E0）。

### 順序の理由

- **E1が先頭**: `best_sizes` が実装済みのため最小コストで「記録更新の快感」を入れられる。他フェーズの成果（ヌシ捕獲・依頼達成）を称号が受け止める土台にもなる
- **E2→E3→E4 の順**: ヌシは既存の出現抽選＋ボス機構の拡張で新画面不要。依頼ボードは新画面を伴うため素材ブリーフ〜visual QAの工程が挟まる。サメはヌシ機構＋ランダムイベント（横取り）が揃ってからが最も安く、環の最終目標として最後に置くのが演出上も正しい
- **E6をE4より前**: サメ海域の解放演出（ボトルメール→海図入手）にE6の仕組みを使うため
- **E7・E8は環の完成後**: §2の方針どおりポリッシュ枠
- **E9が最後**: 全フェーズでカタログ駆動の設計を守れていれば「データ＋素材の差し替えだけで川釣りが成立する」はず。その検証が横展開（別タイトル化）のGo/No-Go判断材料になる

## 4. フェーズ別 詳細計画

### E1. 記録更新演出＋称号

- **目的**: 「前より大きいのが釣れた」を確実に気持ちよくする。釣りゲームの本能的な快感の核
- **仕様概要**:
  - `record_catch()` が自己ベスト更新を検知したら、`CatchFanfare` に「記録更新！ ○○cm（+○cm）」の演出を追加
  - 称号システム: 累計・魚別の実績（例「アジ100匹」「70種コンプ」「全釣り場制覇」）。判定は既存の `caught_counts` / `spot_caught_counts` / `best_sizes` を読むだけの pure 関数として `GameData` 側に置く
  - 称号の表示先はステータス画面（既存の記録セクションに節を追加）
- **接続点**: `PlayerProgress.record_catch()` の戻り値、`CatchFanfare`、`status_screen.gd`
- **実装前の条件**:
  1. `status_screen.gd` が docs/26 R5/R1 の進行対象になっていないことを確認（なっていれば先にそちらを完了）
  2. 称号の獲得済みリストをセーブに追加するため、旧セーブ補完方針を決めてから着手（過去の統計から再計算で復元できる設計を推奨＝セーブ追加を最小化）
- **DoD**: `status_smoke` + `catch_fanfare_smoke` + `save_system_verify.sh` + validate green。ファンファーレの visual 確認

### E2. ヌシシステム

- **目的**: 各釣り場に長期目標を置く。「全釣り場のヌシ制覇」がゲーム全体の中期ゴールになる
- **仕様概要**:
  - 各釣り場に1体、通常個体の2〜3倍サイズのヌシを定義（`FISHING_SPOTS` に `nushi` 節を追加）
  - 出現条件: 特定の天候 × 特定の餌/仕掛け × （E5導入後は時間帯も）。条件成立時のみ低確率で抽選テーブルに乗る
  - 予兆演出: 条件を満たす釣行で「ヌシの気配がする…」等のメッセージ。NPCの目撃情報（港画面の一言）で条件のヒントを出す
  - 捕獲記録はセーブに保存し、魚図鑑に金枠で表示（**決定済み**: ヌシ専用ページは作らず既存ページに金枠バッジ＋ヌシ記録併記）。初回捕獲報酬は `BOSS_FIRST_CLEAR_REWARDS` の機構を流用
  - 図鑑フッターのフィルタタブ列（全魚/港内/砂浜/岩礁/沖/レア）に「ヌシ」フィルタを追加する（ユーザー提案 2026-07-05）
- **接続点**: `game_catalog_data.gd`（FISHING_SPOTS）、`encounter_weights()`、`fish_book_screen.gd`、`PlayerProgress`
- **実装前の条件**:
  1. ~~Fable 単体の設計判断が先~~（決定済み 2026-07-05: 別魚ID方式・`NUSHI_FISH` 辞書新設。詳細は docs/28 §1-2・§E2）
  2. 魚図鑑R5/R1は2026-07-05時点で完了済み。ヌシ表示に入る前に、`docs/qa/fish_book_qa.md` のfreeze表とP2専用魚ポートレート素材待ちだけを再確認する
  3. `docs/qa/fish_book_qa.md` の freeze 値に抵触しないこと（金枠は追加要素であり freeze 値の変更ではない、を確認）。フッターへの「ヌシ」タブ追加はタブ列レイアウトを触るため、freeze 表との照合を先に行い、抵触するならタブ追加の可否を Fable が判断する
- **DoD**: ヌシ出現条件のユニット的検証（headless で抽選監査。`fishing_reveal_smoke` 方式）+ `fish_book_smoke` + 図鑑 visual QA + validate green

### E3. 依頼ボード（注文システム）

- **目的**: 毎回の釣行に「今日の目的」を作る。中盤のダレ（動機のレベル上げ一本化）対策の本命
- **仕様概要**:
  - 港に依頼ボードを新設（例:「アジ5匹」「40cm以上のヒラメ1匹」「刺身用のカツオ」）。同時掲示は3件程度
  - 報酬は お金＋通常入手できない餌・仕掛け・称号
  - 達成判定は納品時（持ち込んだ魚を消費）。`sell_fish` 系のインベントリ操作を流用
  - 依頼の生成はレベル・解放釣り場に応じた重み付きテンプレート方式（手書きリスト＋パラメータ差し込み）
  - リフレッシュは「**依頼条件を達成して帰港した時**」に達成枠を入替（**決定済み** 2026-07-05）。未達成の依頼は掲示され続ける
- **接続点**: 新画面 `src/ui/quest_board_screen.gd`（仮）、`harbor_screen.gd` の導線、`PlayerProgress`（受注状態・達成履歴）
- **実装前の条件**:
  1. **新画面のため `skills/ui-screen-build/SKILL.md` の工程に従う**: 参照画像（`reference/`）の用意 → 素材ブリーフ（docs/22〜24方式）→ 実装 → `docs/qa/quest_board_qa.md` 新設
  2. ~~依頼のリフレッシュ単位の決定~~（決定済み: 達成＆帰港でリフレッシュ）
  3. 受注中依頼のセーブ追加と旧セーブ補完、`save_system_verify.sh` の通過
  4. smoke シーン `tools/quest_board_smoke.tscn` を新設し docs/26 §Smoke 一覧へ追記
- **DoD**: 新smoke + `market_smoke`（インベントリ消費の副作用確認）+ visual QA + validate green

### E6. 釣行中ランダムイベント

- **目的**: 1回の釣行の単調さを崩す。E4の解放演出の下地
- **仕様概要**:
  - 釣行中に低確率で発生: 鳥山（回遊魚の出現率が一時上昇）、流木・ゴミ（ハズレ。ただし稀にアイテム）、ボトルメール（断片を集めると海図→隠し釣り場解放）
  - イベントは抽選と効果を pure 関数化し、演出は釣行画面のメッセージ＋軽いエフェクトに留める（新画面なし）
- **接続点**: `fishing_screen.gd`、`encounter_weights()`（鳥山の一時補正）、`PlayerProgress`（海図断片の保持）
- **実装前の条件**:
  1. `fishing_screen.gd` が docs/26 R1/R5 の進行対象と重なっていないことを確認
  2. 隠し釣り場の実体（釣り場辞書エントリ）を先に定義（E4と共同設計。サメ海域を隠し釣り場にするか否か＝§7-5）
- **DoD**: イベント抽選の headless 監査 + `fishing_reveal_smoke` / `fishing_harbor_return_smoke` 退行なし + validate green

### E4. サメ危険海域

- **仕様概要**:

  - サメ2〜3種を追加（うち1種は deep_ocean 系のヌシ扱い＝E2機構に統合）
  - **リスク設計が本体**: この海域では掛けた魚をサメに横取りされる・仕掛けを切られる（ロスト）イベントが発生する。「行くこと自体が賭け」にする

- **DoD**: `fishing_spot_select_smoke`（ロック・船不足判定）+ 出現監査 + 釣り場マップ visual QA + validate green

### E5. 時間帯（朝まずめ・夜釣り）

- **目的**: 同じ釣り場を別の顔にしてコンテンツを実質倍増する
- **仕様概要**:
  - 時間帯3種（朝まずめ / 日中 / 夜）。天候（`FISHING_ENVIRONMENTS`）と同型の `fish_weight_modifiers` 軸として増設し、抽選は「レベル × 釣り場 × 仕掛け × 天候 × 時間帯」の5軸になる
  - **時間帯は港画面で出港前に選択する**（**決定済み** 2026-07-05・ユーザー提案）。港ヘッダーには既に「潮位・天候・風」の状況行があり、選んだ時間帯もここに表示する。選択UIは「今日の支度」節に置き、食事バフと並ぶ「出港前の意思決定」として統一する
  - 朝まずめはレア率上昇、夜はタチウオ・アナゴ等が活性化。夜釣りは Lv15 で解放
  - 背景・空気感の差し替え素材（時間帯×主要背景）。BGMキーも天候と同様に分岐
- **接続点**: `game_catalog_data.gd`（新テーブル）、`encounter_weights()`、`harbor_screen.gd`（選択UI・ヘッダー表示）、釣行画面の背景解決、E2ヌシ出現条件
- **設計ノート**: 天候は釣行開始時抽選のままなので、「時間帯＝プレイヤーの選択、天候＝自然の運」という対比になる。釣り場マップ画面の「よく釣れる魚」表示を選択中の時間帯に追従させるかは実装時に判断（追従させる場合はマップ画面の freeze 値照合が必要）
- **実装前の条件**:
  1. ~~選択式か抽選かの決定~~（決定済み: 港画面での選択式）
  2. 港画面には `docs/qa/` の freeze ログが現状存在しない（2026-07-05 確認）ため選択UI追加は低リスクだが、変更後は港のスクショ比較を行い、必要なら `docs/qa/harbor_qa.md` を新設する
  3. 背景素材の生成計画（時間帯×釣り場背景の組み合わせ数を先に見積もる。全組み合わせは作らず、色調グレーディングで済む範囲を先に検証）
- **DoD**: 出現監査（時間帯別テーブルの検証）+ `fishing_reveal_smoke` + 港・釣行画面 visual QA + validate green

## 6. 横断事項

1. **抽選ロジックは pure 関数で増設する**: ヌシ条件・依頼生成・イベント抽選・時間帯補正は、docs/26 R3 で分離した `game_catalog_data.gd`（データ）+ `game_data.gd`（pure ロジック）の境界を守って追加する。headless 監査（smoke）で検証できる形を保つ
2. **称号・依頼・図鑑金枠は互いを参照する**: E1称号 ← E2ヌシ捕獲 / E3依頼達成 / E4サメ撃破。称号判定を「統計から再計算する pure 関数」にしておけば、後続フェーズは統計を書くだけで称号が増える
3. **セーブスキーマの追加は各フェーズで最小に**: 統計から導出できるもの（称号など）は保存しない。保存が必要なのは「受注中依頼」「ヌシ捕獲フラグ」「海図断片」「難易度ID」程度に抑える
4. **釣り場マップ画面が混みやすい**: E4（危険海域）・E6（隠し釣り場）・E9（川エリア）はいずれも釣り場マップに追加が入る。freeze 値と R5 の uplift 成果を壊さないよう、マップへの追加は各フェーズでまとめて1回ずつにする
