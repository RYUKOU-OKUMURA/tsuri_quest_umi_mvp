# 釣り場マップUI制作 振り返りレポート

Date: 2026-07-01

対象画面: `src/ui/fishing_spot_select_screen.gd`

このレポートは、釣り場マップ画面を「仮UI」から「80点前後の本番寄り画面」へ引き上げるために効いた判断と、次回以降に同じ品質へ早く到達するための注意点を残す。工程の時系列ではなく、今後のUI制作判断に使う。

## 結論

一番大変だったのは、マップ画像そのものではなく、画面全体の情報設計を完成イメージへ合わせ直すことだった。途中までは下部に釣り場別の達成度セルを並べる方針だったが、正式な完成イメージでは下部は `釣り手帳`、全体達成度、ヒント、`メニュー` の横長フッターだった。ここを理解し直して、釣り場選択の主導線をマップ上のポイントへ一本化したことが、画面の印象を大きく改善した。

もう一つ効いたのは、GDScriptのパネル描画だけで詰めず、木製看板、ステータスバー、フッター、アイコンシートをPNG素材として生成して使ったこと。水中ファイト画面と同じく、完成度が必要な部位は素材化し、Godot側は状態、文字、選択、ロック、payloadを担当する分担にしたことで、汎用UIから本番画面らしい見た目へ近づいた。

## 何が大変だったか

### 正式な完成イメージが途中で更新された

当初は、下部に釣り場別セルを並べる「達成度台帳」を磨く方向で進めていた。しかし正式な完成イメージでは、そのセル一覧は存在しなかった。下部セルを残したまま装飾だけ改善しても、ユーザーが求めている「マップ上で釣り場を選ぶ体験」と矛盾する。

大変だった点:

- 既に作ったUIを磨くのではなく、構成そのものを捨てる判断が必要だった。
- `docs/15_fishing_spot_encounter_spec.md` の後続解釈と、正式画像のUI意図を合わせる必要があった。
- smoke test も「8セルがあること」ではなく「8セルが残っていないこと」を検証する形へ変える必要があった。

今後は、見た目を詰める前に「正式画像で何が存在し、何が存在しないか」を明文化する。

### マップの細部より、外枠と情報階層の差が大きかった

途中ではマップ画像のディテールやポイントアイコンの改善へ意識が寄りがちだった。しかし完成イメージとの差分で大きかったのは、上部看板、右上ステータスバー、右詳細パネル、下部フッターの構成だった。

大変だった点:

- マップ自体は既に一定品質に達していたため、細部を触っても画面全体の印象は大きく変わらなかった。
- 下部が釣り場選択リストに見えると、マップ上のポイントを選ぶ魅力が弱くなる。
- 右詳細が細かい行と小さいボタンに分かれていると、完成画像の「ここで釣る」主導線に見えない。

今回の改善では、地図アートよりもUI枠の構成に集中したことが効いた。特に、下部を4要素の操作フッターに作り替えたことで、画面の読み順がかなり整理された。

### Godotの実表示でしか分からない文字のはみ出しが多かった

実装中は、コード上の `size` や `custom_minimum_size` だけでは判断できないはみ出しが出た。特に下部の `メニュー`、右詳細の魚名、右詳細下部の `港へ戻る`、タイトル看板の文字重なりは、スクショを見て初めて分かった。

大変だった点:

- 日本語の文字幅は、想定より早く省略表示になりやすい。
- アイコン付きボタンは、Button標準の `icon` と `text` 任せだと調整しにくい。
- 右詳細は縦幅が限られており、サムネイルを大きくすると下のボタンが見切れやすい。

対応として、左右ボタン内に `TextureRect` と `Label` を自前配置し、右詳細はサムネイル、行高、ボタン高を少しずつ圧縮した。文字は `clip_text` と `text_overrun_behavior` を入れたうえで、実スクショで見切れがないことを確認した。

### 新規PNG素材の読み込みとimport状態に注意が必要だった

新しく追加した `map_title_sign.png`、`map_status_bar.png`、`map_status_icon_sheet.png`、`map_footer_icon_sheet.png` は、Godotの `.import` が揃う前だと `ResourceLoader.exists()` だけでは実行環境によって読めない可能性がある。

大変だった点:

- プレビューでは素材が見えているつもりでも、実際にはフォールバック表示になっている場合がある。
- 新規素材は `.import` 生成前後で挙動が変わる。
- スクショ上で「看板の文字がない」「アイコンがない」と見えたとき、原因がレイアウトなのか読み込みなのか切り分けが必要だった。

対応として、`_load_texture_if_exists()` に `ImageTexture.create_from_image()` のフォールバックを入れ、新規PNGを直接読めるようにした。これで headless preview でも素材表示が安定した。

### UI変更と釣行継続ロジックがつながっていた

今回の主題は見た目だったが、`釣り場を変える` 導線は `trip_stats` の扱いにも影響した。釣り場変更だけで食事バフや釣行ステータスを再消費しないためには、釣り場選択画面から釣り画面へ戻るときに既存 `trip_stats` を引き継ぐ必要がある。

大変だった点:

- 画面だけ完成しても、継続釣行で `spot_id` が正しく更新されないと仕様として壊れる。
- `continue_trip` では `trip_stats` を再計算しない一方、選択した釣り場情報だけは更新しなければならない。
- smoke test で初めて、継続釣行時に `outer_tide` がレベルゲートで弾かれて default に戻る問題が見つかった。

対応として、`continue_trip` のときはマップ側で解放済み選択を済ませている前提で `spot_id` を信頼し、`_apply_spot_to_trip_stats()` で `spot_id`、釣り場名、水深、対象魚、推奨エサだけを上書きするようにした。

## 何を意識したら80点品質に届いたか

### 1. 参照画像の構成を「存在する要素」と「存在しない要素」に分けた

今回の大きな転換点は、下部8セル台帳を磨くのではなく、正式画像に存在しないものとして撤去したことだった。

意識したこと:

- 上部は「木製タイトル看板 + 1本のステータスバー」にする。
- 下部は「釣り手帳 / 達成度 / ヒント / メニュー」にする。
- 釣り場選択はマップ上のポイントに一本化する。
- 右詳細は釣り場名、サムネイル、3行情報、主ボタン、副ボタンに絞る。

この整理がないまま装飾だけ足しても、正式画像のUXには近づかなかった。

### 2. UI枠をPNG素材として作った

コード描画だけでは、木、紙、濃紺バー、金具、古い航海図のような質感が出にくい。今回追加した素材は、画面の第一印象を作る役割に絞った。

追加した主な素材:

- `map_title_sign.png`
- `map_status_bar.png`
- `map_status_icon_sheet.png`
- `map_footer_frame.png`
- `map_footer_icon_sheet.png`

日本語テキストは素材に焼き込まず、Godot側で描画した。これにより、データ変更、翻訳、フォント調整、状態変更に耐えられる。

### 3. マップそのものを触りすぎず、画面の読み順を直した

ユーザーからの指摘通り、今回の品質差分はマップ内画像のディテールではなく、上部、下部、右詳細の設計にあった。そこで、マップ背景の描き込みよりも次を優先した。

- 上部タイトルの視認性
- 右上ステータスバーのまとまり
- 右詳細の主ボタンの強さ
- 下部フッターの要素数削減
- 釣り場選択の操作対象をマップに限定すること

この優先順位にしたことで、画面全体が「情報カードの集合」ではなく「航路図から出航先を選ぶ画面」として読めるようになった。

### 4. 実スクショを毎回見て調整した

`/tmp/tsuri_fishing_spot_map.png` を出して、正式画像と比較したことが有効だった。コード上では収まっているように見えても、スクショでは文字の重なり、余白の穴、右詳細の見切れ、ボタン内テキストの省略がすぐ見えた。

今回スクショで見つけて直したもの:

- タイトル看板上で錨と文字が重なる。
- 新規素材が表示されず、看板が空に見える。
- 右詳細の `港へ戻る` が下で見切れる。
- 下部右の `メニュー` が省略表示になる。
- フッター中央右に余白の穴が見える。

今後も、UI品質作業では実スクショなしで完了判断しない。

### 5. smoke test をUX契約に合わせて更新した

以前の smoke は下部に釣り場別達成度セルがあることを前提にしていた。正式画像ではそれが誤りになったため、テストも更新した。

更新した検証観点:

- フッターに全体達成度ラベルが存在する。
- 下部に `spot_progress_entry` の釣り場セルが存在しない。
- 釣り場カードボタンが下部に残っていない。
- 継続釣行で `trip_stats` が引き継がれる。

見た目の変更でも、テストが古いUXを正としたままだと次回また戻ってしまう。今回のように、テストも完成イメージの契約へ寄せる必要がある。

## まだ80点に留まっている理由

今回の画面は、仮UIからは大きく改善したが、正式画像に対してまだ差分は残っている。

残っている主な差分:

- マップ上のポイントラベルと航路線は、正式画像よりややゲーム内データ表示寄りで、絵としての統一感はまだ弱い。
- 木製看板とステータスバーは素材化したが、正式画像ほどの厚み、ロープ、陰影、文字印字感はない。
- 右詳細の釣り場サムネイルは既存マップ切り抜きで、正式画像のような専用イラスト感は弱い。
- 達成度は実データ集計を優先しており、正式画像の `2/20` のような見せ方とは完全一致していない。
- 画面全体のフォントはまだ共通UI寄りで、完成画像の重い見出し文字、読みやすい本文、アイコン付き行の階層感には届いていない。

今回の到達点は、構成とUX契約を正式画像へ寄せた80点品質。90点以上へ上げるには、さらに本番素材と専用タイポグラフィが必要になる。

## 次に90点へ上げるなら

### 1. 右詳細サムネイルを専用素材化する

現在はマップ背景から各釣り場のサムネイルを切り出している。90点を狙うなら、各釣り場の特徴が一目で分かる専用サムネイルを作る。

優先順:

- `harbor_pier`: 港、堤防、小舟
- `shallow_sand`: 砂浜、浅瀬、かけあがり
- `rock_breakwater`: 消波ブロック、根魚の気配
- `outer_tide`: 潮目、流れ、回遊魚

右詳細は選択中ポイントの魅力を伝える場所なので、ここが強くなると画面全体の完成度が上がる。

### 2. マップポイントを正式画像寄りの大きなメダルにする

現状のポイントアイコンは機能的には十分だが、正式画像のような「触りたい目的地」にはまだ弱い。

改善案:

- 解放済みポイントは大きめの円形メダルにする。
- 選択中ポイントは発光、太い金縁、接続線で主役化する。
- 未解放は小さな鍵だけでなく、暗いメダルと必要Lvプレートを一体化する。
- ラベルをマップ背景に埋め込まず、ポイントメダルの付属プレートとして読む。

マップ選択が主導線なので、ポイント自体の魅力は次の大きな改善対象になる。

### 3. フッターの達成度仕様を釣り手帳と接続する

ユーザー意図として、下部の達成度は将来的に「その釣り場で釣った魚のコンプリート状況」や釣り手帳に紐づく。今回の実装では全体達成度を表示しているが、次は手帳仕様と接続して意味を強める。

検討事項:

- 全体達成度なのか、選択中釣り場の達成度なのかを明確にする。
- `2/20` の denominator をどの魚種集合にするか決める。
- 釣り手帳ボタンから確認できる情報と、マップ下部に出す情報を分離する。
- ヒント欄は「未記録魚」「おすすめエサ」「回遊ヒント」のどれを優先するか決める。

ここは見た目だけでなく、ゲーム体験の報酬設計に関わる。

### 4. 専用フォントと見出し文字を検討する

正式画像では、見出しとボタン文字が画面の印象を強く作っている。現状は共通テーマの文字に寄っているため、次は文字の質感を上げる。

改善案:

- 木製看板のタイトルだけ太めの専用見出しフォントにする。
- 右詳細の釣り場名は濃紺帯に合わせた大きい白文字へ固定する。
- 本文、数値、ボタンでフォントサイズと太さを分ける。
- 小さい日本語は無理に装飾せず、読みやすさ優先にする。

フォントを一括置換すると崩れるため、見出し、数値、本文、ボタンに分けて適用する。

### 5. 比較用プレビューを正式QAツール化する

今回 `/tmp/tsuri_fishing_spot_map.png` を毎回確認したのは有効だった。次はこれを QA として固定する。

追加したい確認:

- 1280x720で文字が省略されていない。
- 下部に8セル台帳が存在しない。
- 上部に木製看板と1本のステータスバーが存在する。
- 右詳細の主ボタン、副ボタンが見切れていない。
- 新規PNG素材が実表示されている。

目視だけでなく、代表ノード名やメタ情報で最低限の構成を検出できると回帰に強くなる。

## 次回以降の実装ルール

### 1. 正式画像を先に分解する

実装前に、正式画像を次の観点で分解する。

- 存在する領域
- 存在しない領域
- 主操作
- 補助操作
- データ表示
- 画像素材で担うべき部分
- Godot runtimeで描くべき部分

特に「存在しない領域」を明記する。今回の下部8セル台帳のように、作り込むほど完成イメージから離れる要素がある。

### 2. 構成を合わせてからディテールを触る

マップの波、島の描き込み、ポイントアイコンの細部より先に、上部、右詳細、下部の構成を合わせる。構成がズレている状態で細部を磨くと、労力の割に完成画像へ近づかない。

優先順位:

1. 画面領域の構成
2. 主導線
3. 情報階層
4. 文字のはみ出し
5. 素材の質感
6. 細部の演出

### 3. UI枠は早めに素材化する

本番寄り画面では、汎用 `PanelContainer` と `StyleBoxFlat` だけでは限界が早い。木、紙、金具、濃紺バー、アイコンはPNG素材にして、Godot側は文字と状態に集中する。

ただし、テキストは基本的に素材へ焼き込まない。釣り場名、Lv、所持金、達成度、説明文はデータで変わるため、runtime描画にする。

### 4. 実スクショを完了条件に入れる

UI品質作業では、テストが通っても見た目が崩れていることがある。`/tmp/tsuri_fishing_spot_map.png` のような固定スクショを必ず出し、参照画像と並べて見る。

見るべき項目:

- 文字が見切れていないか
- 余白に不自然な穴がないか
- 主ボタンが一番目立つか
- 選択操作がどこか一目で分かるか
- 画面の上部、右、下部が同じ世界観に見えるか

### 5. 見た目の変更でも既存フローを必ず検証する

今回のように、釣り場変更導線は見た目だけでなく `trip_stats`、食事バフ、釣行継続に影響する。UI変更でも次は必ず確認する。

- 釣り場選択 smoke
- 魚名非表示 smoke
- 港戻り smoke
- 釣り場別出現監査
- Godot validate

特に `釣り場を変える` は、港へ戻る導線と役割が違う。釣行終了ではなくポイント変更なので、食事バフや釣行ステータスを再消費しないことを毎回守る。

## 今回の成果

今回の実装で到達した状態:

- 上部が木製タイトル看板と濃紺ステータスバーになった。
- 下部から8セル釣り場台帳を撤去し、操作フッターへ置き換えた。
- 右詳細を羊皮紙パネル、濃紺タイトル、サムネイル、3行情報、主/副ボタンへ整理した。
- 釣り場選択の主導線をマップ上のポイントに一本化した。
- 継続釣行で `trip_stats` を再利用し、釣り場情報だけ更新するようにした。
- `/tmp/tsuri_fishing_spot_map.png` を使って実スクショ比較した。
- smoke test、釣り場別出現監査、Godot validate を通した。

今回の品質は、本番最終アートではないが、仮カードUIからは明確に抜け出した。次の品質向上は、コード合成の微修正ではなく、右詳細サムネイル、マップポイントメダル、専用フォント、釣り手帳連動達成度の4点を素材/仕様フェーズとして扱う。
