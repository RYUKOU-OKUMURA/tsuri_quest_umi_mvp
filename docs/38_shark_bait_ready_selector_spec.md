# 38. サメ餌魚のREADYセレクタ＋レア度耐久チャージ 実装仕様

作成日: 2026-07-07
状態: 実装完了（2026-07-07）
位置づけ: `docs/37_shark_bait_in_trip_management_design_note.md` の全面採用を前提に、(1) レア度別の耐久チャージ、(2) READY画面の餌魚セレクタUI、(3) docs/37レビューで出た4つの穴（経済バランス・奇襲プラン・抽選タイミング統一判断・継続釣行フォールバック）を確定させた実装仕様。実装はこのdocとdocs/37を正本とする。

## 0. 要約

- 危険海域の釣行中、READY状態で餌魚を選び、`投げる` 押下時に消費する（docs/37どおり）
- 新規: 餌魚は**レア度に応じたキャスト回数（チャージ）**を持つ。1匹消費すると、コモン1回／アンコモン2回／レア3回／ぬし5回のキャストに使える
- チャージは釣行内のみ有効（`trip_stats` 管理・非セーブ）。餌魚を切り替えても釣行内なら残チャージは保持される
- 抽選（`roll_hooked_fish`）はキャスト押下時へ移動する。**v1では危険海域のみ**（抽選遅延フラグで実装し、通常釣り場・ボス釣り場は現行タイミングを維持）
- READY下段HUDを専用バーへ切り替え、左に餌魚セレクタ、中央に大型 `投げる`、右に釣り場変更/港戻りを置く。CASTING以降の現行ゲージHUDは維持する

## 1. 経緯と決定

- 2026-07-07: docs/37（READY選択＋キャスト時消費）を全面採用とユーザー決定
- 2026-07-07: ユーザー提案「レア度によるロストしにくさ」を採用。docs/36不採用表の「1投ごと消費はボウズ損失が子供に刺さる」「メガロドン抽選は釣行単位前提」という残論点を、レア度チャージで解消する
  - コモン（アジ等・売値〜360G）は1回で消費: 損失が軽く「投げたら減る」が明快
  - ぬし級（売値1,680〜14,400G）は5回: メガロドン狙いの1キャストコストが約340〜2,880Gに収まり、毎キャスト全損（最大14,400G/キャスト）を回避
- 2026-07-07: 抽選タイミングは危険海域のみキャスト時へ移動と決定。全釣り場統一案は、READY中のタナ表示（抽選魚の `start_depth` 由来）が通常釣り場で変わる回帰と、`fishing_harbor_return_smoke` 全域の改修コストに見合わないため不採用

## 2. 耐久チャージ仕様

### 2-1. レア度→チャージ数

正本は `GameData` に新設する変換（例: `shark_lure_charges_for(fish: Dictionary) -> int`）。判定は `nushi` フラグまたは `rarity` 文字列で行い、売値では段階化しない。

| rarity | チャージ数 | 該当餌魚候補数 | 売値レンジ |
|---|---|---|---|
| コモン | 1 | 18種 | 85〜360G |
| アンコモン | 2 | 20種 | 260〜1,500G |
| レア | 3 | 31種 | 680〜3,600G |
| ぬし | 5 | 8種 | 1,680〜14,400G |

補足:

- `boss_kurodai` は `rarity == "ぬし"` なのでチャージ5。通常ヌシ魚は現データでは `nushi: true` かつ `rarity == "レア"` のため、`nushi` フラグを優先してチャージ5にする。メガロドンのヌシ級述語には非該当のまま（述語は変更しない）
- キハダ等「レアだが売値3,000G以上」はチャージ3のままメガロドン餌に使える。ぬし個体より消費が速い分、入手しやすさで釣り合う

### 2-2. 消費とチャージのフロー

`投げる` 押下時（危険海域・READYのみ）:

1. 選択中餌魚が `餌魚なし` → 消費せず、餌魚重みなしで抽選へ
2. 選択中餌魚に釣行内の残チャージがある → チャージを1減らし、その魚を有効餌魚として抽選へ。**インベントリは減らない**
3. 残チャージがなく、所持数 > 0 → `PlayerProgress.consume_fish_for_shark_lure()` で1匹消費し、`チャージ数 - 1` を残チャージとして記録（今回のキャストで1回分使用）。その魚を有効餌魚として抽選へ
4. 残チャージがなく、所持数も 0 → キャストしない。READYのまま「餌魚がありません」系メッセージを出し、選択を `餌魚なし` に戻す

原則の言い換え: 「投げたら1匹減る」は「チャージがない魚を投げたら1匹減って、レア度の回数分投げられるようになる」に拡張される。UIで残回数を常時見せることで明快さを保つ（§4）。

### 2-3. チャージの寿命と保持

- 置き場: `trip_stats["shark_lure_charges"]`（`fish_id -> 残回数` のDictionary）。`trip_stats` は非セーブ・釣行単位なので、**港帰還・釣行終了で残チャージは消える**
- 釣行継続（釣り場変更）では `trip_stats` の複製引き継ぎに乗って残チャージも維持される
- **餌魚を切り替えても残チャージは捨てない**。ぬし魚を1回使ってからアジを試し、またぬし魚に戻る、という実験を罰しないため。UIには選択中の魚の残回数だけを出し、全魚のチャージ一覧は見せない（帳簿化の禁止）
- 所持数0でも残チャージがある魚はセレクタ候補に残す（例: 最後のぬし魚を消費済みで残4回）。チャージ0になったら候補から外れる
- ファイト中断・逃げられた場合もチャージ・消費は戻らない（docs/37 §3-5の原則を踏襲）
- 港帰還時に残チャージがある場合、v1では警告を出さない（釣行を終える操作自体が明示的なため）。遊んでみて損失感が強ければ帰還確認に1行足す（未決事項ではなく将来の調整枠）

## 3. 抽選タイミング（docs/37 §4の確定）

### 3-1. 実装形

`_prepare_new_attempt()` を「リセット」と「抽選＋prepare」に分割し、抽選遅延フラグで切り替える。

- 通常釣り場・ボス釣り場: 現行どおり挑戦準備時に抽選（ボスは固定魚）。挙動・smoke非変更
- 危険海域: 挑戦準備時は抽選せず、`_simulator.prepare({})`（プレースホルダ）でREADYに入る。`投げる` 押下時に §2-2 の消費フロー → `roll_hooked_fish()` → `_simulator.prepare(抽選魚, trip_stats)` → `_simulator.cast()` を同一ハンドラで連続実行する

### 3-2. 一緒に動かすもの（危険海域のみ）

- `_prepare_shark_ambush_plan()`: 抽選魚に依存するため、キャスト時の抽選直後へ移動
- `bird_swarm_hits_remaining` の重み適用・減算: 抽選と同時（キャスト時）へ移動。「WAITINGで鳥山付与→次キャストから適用」の1キャスト遅れ構造は現行と同型のため仕様として維持
- `fight_sidebar.bind` / `fight_hud.bind` / ビューのbind: READY時は空魚データでbindし、キャスト時に抽選魚で再bindする

### 3-3. READY中の未抽選表示（危険海域）

- 魚影カード: 既存の「未確認の魚影」パスを使う。ただし `preferred_bait` フォールバックと boss 判定は空dictで安全に落ちることを確認する
- タナ表示（HUD・サイドバーとも `simulator.depth` 由来）: 未抽選中は `タナ：--` を出す。プレースホルダprepareのデフォルト深度（8m）を実値のように見せない
- 魚素材の事前ロードはキャスト時抽選後に行う（READY中は対象なし）

### 3-4. 継続釣行フォールバック

`fishing_spot_select_screen.gd` が `_trip_stats["shark_lure_fish_id"]` を選択フォールバックに使う現行処理（L95-99）は、新semanticsでは「前回キャストで消費済みの有効餌魚」を読むことになる。これは**次の初期選択として流用してよい**（同じ魚を続けて投げたい §3-4 docs/37 と同義）。ただし消費済み扱いにはせず、釣行画面入場時に画面ローカルの `_selected_shark_lure_fish_id` の初期値としてのみ使う。route payload経由の港側初期選択も同様（docs/37 §5-1の8）。

## 4. READY画面UI仕様（2026-07-07改訂: READY専用下段バーへの刷新）

参照画像: `reference/13_fishing_ready_danger_mockup.png`（2026-07-07生成。方向性の真実であり最終アートではない）

本節は基盤レイアウト原則（`docs/19` §4.6、docs/30 決定#19）のREADY状態への適用である。FIGHT状態への適用は `docs/39_underwater_fight_ui_redesign_spec.md`（本docの完了後に着手）。下段バーの枠素材・高さ設計はdocs/39と共通化するため、READY実装時にFIGHT側の3ゾーン構成（docs/39 §3）を一読しておくこと。

当初案（左下パネルの中身差し替えのみ）は、READY画面全体のクオリティを上げたいというユーザー決定により、**READY状態専用の下段バー刷新**へ拡張された。旧案は本節の改訂で上書きする。

### 4-1. READY専用下段バーの構成

READY状態では、下段HUD（高さ224px枠は維持）を専用レイアウトに切り替える。CASTING以降は現行のゲージHUD（テンション／タナ／魚の体力＋操作ヒント）へ戻す。

READY中にゲージを消す根拠:

- テンション・魚の体力はREADY中は無意味（魚が掛かっていない）
- 現行のREADYタナ表示は抽選済み魚の `start_depth` を表示しており、正体秘匿の観点でも情報漏れだった。§3の抽選遅延（危険海域）とも整合する

3ゾーン構成（参照画像どおり）:

| ゾーン | 内容 |
|---|---|
| 左（約40%） | 餌魚セレクタ（§4-2）。通常釣り場では現行の「使用中のエサ」情報（仕掛け名＋対応餌カテゴリ）を同じカード様式で表示し、矢印は出さない |
| 中央 | 大型の主操作ボタン `投げる`＋`E / Enter` キーチップ。現行の「操作ヒント欄の1項目」から画面の主役へ格上げする |
| 右 | `＋ 釣り場変更` / `− 港へ戻る` の縦2ボタン（現行機能・キー割当を維持） |

### 4-2. 餌魚セレクタ（危険海域READYのみ）

`◀` `▶` 矢印＋中央カードの構成。カードには以下を出す。

- 見出し: `サメ餌魚`
- 魚ポートレート: `FightFishAssets` 経由（直接パス参照禁止）
- 魚名＋所持数（例: `アジ x12`）
- チャージピップ＋残回数: ◆◆◇ `あと2回`（チャージ保有時）／ `投げると1匹つかう`（未消費時）
- 状態別の表示内容:

| 状態 | 主表示 | 補助行 |
|---|---|---|
| チャージあり | アジ ＋ ピップ | あと2回 |
| チャージなし・所持あり | アジ x12 | 投げると1匹つかう |
| 餌魚なし | 餌魚なし | 通常サメ狙い |
| 所持0・残チャージあり | 深淵の重鎮 ＋ ピップ | あと4回（在庫0） |

CASTING以降は下段が現行ゲージHUDへ戻るため、消費済みの有効餌魚の表示は従来どおり左下「使用中のエサ」パネル（`餌魚：<魚名>`。docs/36改修A）が担う。残回数を添える（例: `餌魚：アジ（あと2回）`）。「READY＝未消費の選択」「CASTING以降＝消費済みの今回の餌魚」の読み分け（docs/37 §6-2）は維持。

### 4-3. 操作

- `◀` `▶` 矢印クリックで前後の候補へ（`fight_hud.gd` に `Rect2` ヒットテスト＋シグナル追加。既存の `_main_rect` 等と同じパターン）
- キーボード: READY時のみ `←` / `→` で前後送り（E/Enter・Space・Shift・+/-・Escと衝突しない）
- 候補列挙: `所持数 > 0` または `釣行内残チャージ > 0` の非サメ魚。順序はインベントリ順を基本とし、先頭は `餌魚なし`
- 変更可能はREADYのみ。CASTING以降は矢印・キーとも無効（docs/37 §3-5）
- 大きな一覧パネル・ソート・フィルタ・好物表示は入れない（docs/37 §6-3、§8リスク4）

### 4-5. freeze・QAとの関係

- `docs/qa/fishing_surface_qa.md` の「サメ餌魚HUDは在庫・残数を表示しない」freezeは、docs/36の消費モデル前提が本docで消えるため改定する（判断ログ必須）
- `docs/qa/underwater_fight_qa.md` のHUD freeze（高さ224px・下段幅比率26.5%/17.5%・ゲージ18分割）は**FIGHT系状態の値として維持**する。READY専用バーは新レイアウトのため既存freezeの対象外だが、外枠高さ224pxは共有し、READY用のfreeze値は実装後にQAへ新設する
- 通常釣り場のREADYも下段バー刷新の対象になる（ゲージ非表示＋左ゾーンがエサ情報カード）。WAITING〜FIGHT・釣果は非変更。通常釣り場READYの前後比較スクショを必ず残す

## 5. 状態モデル

| 状態 | 置き場 | 意味 | 寿命 |
|---|---|---|---|
| `_selected_shark_lure_fish_id` | `fishing_screen` ローカル | READYで次に投げたい魚。未消費 | 画面内（入場時にroute/trip_statsから初期化） |
| `trip_stats["shark_lure_fish_id"]` / `..._name` | trip_stats | 今回のキャストで消費（またはチャージ使用）された有効餌魚。抽選・HUD・文言・ファンファーレが読む | キャスト〜次のREADY |
| `trip_stats["shark_lure_charges"]` | trip_stats | `fish_id -> 残回数`。釣行内の耐久チャージ | 釣行内（港帰還で破棄） |

- READYで選び直しても `trip_stats["shark_lure_fish_id"]` は更新しない（docs/37 §5-2の不変条件）
- 有効餌魚は毎キャストの消費フロー内で設定し、キャストしなかった場合は前回値のまま触らない
- 新しいREADYに入ったとき（`_retry`）、有効餌魚表示はREADYセレクタ表示に切り替わるため、古い有効餌魚が「今回の餌魚」として見え続けることはない

## 6. 港・釣り場選択の扱い

docs/37 §3-6の推奨どおり。

- `fishing_spot_select_screen.gd` `_select_spot()` の消費処理を撤去。payloadには選択IDだけ載せる
- 港画面「今日の支度」の餌魚ボタンは**初期選択**として残す（消費しない）。ボタン下の説明を「危険海域で投げるときに使う」旨に更新
- `_apply_route_shark_lure_to_trip_stats()` は廃止し、route payloadの餌魚IDは `_selected_shark_lure_fish_id` の初期値としてのみ扱う

## 7. 検証方針

docs/37 §7に以下を追加・上書きする。

チャージ関連:

- コモン魚は1キャストで所持数が1減り、次キャストで再度1減る
- ぬし魚は1キャスト目で所持数が1減り、2〜5キャスト目は減らない。6キャスト目で再度1減る
- 餌魚を切り替えて戻しても残チャージが保持される
- 所持0・残チャージありの魚が候補に残り、チャージ0で候補から消える
- 港帰還→再出港で残チャージが消えている
- 残チャージ使用キャストでも有効餌魚として重み補正・好物ファンファーレ・メガロドン述語が機能する

抽選タイミング関連（調査済みの影響先）:

- `fishing_harbor_return_smoke`: 危険海域の節（L225-297相当）のみ「READYで魚未確定→キャストで確定」前提に更新。デフォルト釣り場の節は非変更
- `fishing_spot_select_smoke` / `harbor_screen_smoke`: 釣り場選択・港で消費しない前提へ更新
- `fishing_surface_states_preview`: 危険海域READYの未抽選表示（タナ：--）のキャプチャを追加
- `shark_lure_audit` / `shark_ambush_audit` / `fishing_reveal_smoke` / `catch_fanfare_smoke`: 非変更（データ層・simulator単体のため）。`shark_lure_audit` の期待値が変わらないことを回帰確認に使う
- 通常釣り場のREADYタナ表示・鳥山適用が現行から変わらないこと（分岐漏れの検出）

UI:

- 危険海域READYの下段バー（セレクタの§4-2状態別表示を含む）を実スクショ＋`reference/13_fishing_ready_danger_mockup.png` との横並び比較で確認し、`docs/qa/fishing_surface_qa.md` と `docs/qa/evidence/fishing_surface/` に残す
- 通常釣り場READYの下段バー（ゲージ非表示＋エサ情報カード＋大型投げるボタン）の前後比較スクショを残す
- WAITING〜FIGHT・釣果画面の下段HUDが現行から変わらないこと（リグレッション比較。`underwater_fight_qa.md` のfreeze値維持の確認を含む）
- CASTING以降の左下「使用中のエサ」パネルに有効餌魚＋残回数が出ること

## 8. リスク

docs/37 §8に加えて:

- **リスク5: チャージの帳簿化** — 全魚の残チャージ一覧を見せたくなるが、見せるのは選択中の魚の残回数だけに固定する。一覧要求が出たら本docの§2-3を改訂してから
- **リスク6: 分岐漏れ** — 抽選遅延は危険海域のみのため、通常釣り場の経路に手が入らないことをsmoke（§7）で担保する。実装では遅延フラグを1箇所（釣り場データまたは `_spot_id == "danger_reef"` 判定のヘルパ）に集約し、画面コードに条件を散らさない
- **リスク7: READYプレースホルダの露出** — `prepare({})` のデフォルト値（深度8m等）が実値のように表示される事故。READY中はゲージ非表示（§4-1）とサイドバーのタナ未抽選分岐（§3-3）で塞ぎ、スクショで確認する
- **リスク8: READY刷新の回帰範囲** — 下段バー刷新は通常釣り場のREADYにも及ぶ。FIGHT系HUD・釣果演出には一切触らないことをbrief側で明示し、状態遷移ごとの前後比較スクショで担保する
